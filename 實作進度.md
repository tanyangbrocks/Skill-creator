## 程式碼待辦（Code Review 發現）

### 🔴 Bug / 正確性（已全部修完 ✅）

**B-1｜`If` 積木的 `ElseBranch` 在 UI 完全不可用** ✅
- 位置：`ScratchCanvas.cs` `BuildBlockCard`（只渲染 `ThenBranch`，未渲染 `ElseBranch`）
- 問題：`SpellCompiler` 已正確編譯 `ElseBranch` 的跳轉邏輯，但使用者無法從 UI 加入積木到 else 分支，`If` 事實上變成沒有 else 的條件。
- 修法：在 `BuildBlockCard` 的 `If` 判斷區新增 `BuildBranch(block.ElseBranch, "ELSE", indent + 1)`。

**B-2｜`SpellCompiler.EmitBlock` 靜默丟棄未知 `BlockType`** ✅
- 位置：`SpellCompiler.cs` `EmitBlock`（switch 無 `default` 分支）
- 問題：新增 `BlockType` 若忘記加 compiler case，積木在執行時無聲消失，沒有任何錯誤或警告。
- 修法：加入 `default: GD.PushWarning($"[SpellCompiler] 未處理的 BlockType: {block.Type}");`

**B-3｜`EntityInfo.Hp` 在 `ForEachNearby` 迭代中陳腐** ✅
- 位置：`ExecutionLoop.cs` `ForEachStart` / `ForEachStep`（`EntityInfo` 為施放時快照）
- 問題：`SetEntityProp` 扣血後實際 `Enemy.Hp` 更新，但 `CurrentIterEntity.Value.Hp` 和 `_e.hp` 變數仍是舊值。同次迭代呼叫 `GetEntityProp "hp"` 會讀到錯誤數值，依此計算的後續 `SetEntityProp` 也會錯。
- 修法：`SpellCaster` / `SpellRunner` 消費 `PendingEntityDamageIdx` 後，同步更新 `ctx.CurrentIterEntity` 的 Hp 欄位（需重建 record）並重寫 `_e.hp`。

**B-4｜`GlobalVars` / `GlobalLists` 從不重置** ✅
- 位置：`ExecutionContext.cs`（兩個 `static Dictionary`）；`EventBus.cs`（`static HashSet _active`）
- 問題：這三個靜態集合在遊戲重啟或場景重載時不會清除。同一 process 跑第二局，全域變數帶著上局狀態。`EventBus` 有 `ClearFrame()` 但沒有 session-level reset。
- 修法：在 `Main._Ready()` 加入 `ExecutionContext.GlobalVars.Clear(); ExecutionContext.GlobalLists.Clear(); EventBus.ClearAll();`（需在 EventBus 新增 `ClearAll` 方法）。

**B-5｜`MaxComboDepth = 5` 重複定義** ✅
- 位置：`SpellCaster.cs:11` 與 `SpellRunner.cs:23` 各有一份
- 問題：若只改一處，兩條執行路徑的連段上限悄悄走岔。
- 修法：移至 `SafetyGuard.cs` 作 `public const int MaxComboDepth = 5;`，兩處改為引用。

### 🟡 設計 / 技術債（已全部修完 ✅）

**D-1｜`SpellCaster.ExecuteEffects` 與 `SpellRunner.Advance` 高度重複** ✅
- 位置：`SpellCaster.cs`（`ExecuteEffects` 方法）、`SpellRunner.cs`（`Advance` 方法）
- 問題：兩者都做「建 ctx → 驅動 loop.Step → 消費 PendingInvokeTotem / InvokeSpell / EntityDamage → 處理 ForEach 位置覆蓋」。每次新增 pending 機制（目前三種），兩個地方都要同步修改。
- 修法：提取 `static void AdvanceContext(ExecutionContext ctx, ExecutionLoop loop, ContextRuntime rt, float delta)` 共用方法，`rt` 封裝 player / world / enemies / loadout 等依賴。

**D-2｜`ForEachNearby` 的 `player.Position` 暫時覆蓋** ✅
- 位置：`SpellCaster.cs` 與 `SpellRunner.cs` 的 `PendingInvokeTotem` 處理段
- 問題：用 save/restore 手法改寫共享物件欄位。目前 `SpellRunner` 的多個 `ActiveSpell` 是同幀序列執行，不存在競態。但若未來加入真正並行（Task / async），此處是 race condition 的埋雷點。
- 修法：在 `ExecutionContext` 加入 `GridPos? EffectOriginOverride`，讓 `SpellCaster.ResolveTotem / ExecuteSlot` 讀取此欄位決定效果中心，不再直接修改 `player.Position`。

**D-3｜安全迴圈魔法數字 `300`** ✅
- 位置：`SpellCaster.cs:115` (`while (!ctx.IsFinished && safety++ < 300)`) 與 `SpellRunner.cs:98`（相同）
- 問題：`300` 散落兩處，語義不明。
- 修法：加入 `SafetyGuard.MaxStepsPerCast = 300`，兩處改為引用。

**D-4｜`QueryEnemies` 重複計算 `DistanceTo`** ✅
- 位置：`SpellCaster.cs` `QueryEnemies`（`Where` 與 `OrderBy` 各算一次，含 sqrt）
- 問題：每個敵人被計算兩次距離。對象少時無感，若敵人數量增加會浪費。
- 修法：改為 `.Select((e, i) => (e, i, dist: e.Position.DistanceTo(player.Position))).Where(x => x.dist <= radius).OrderBy(x => x.dist).Select(x => new EntityInfo(x.i, x.e.Position, x.e.Hp, x.e.MaxHp))`。

**D-5｜`BroadcastAndWait` 行為與 `Broadcast` 完全相同** ✅
- 位置：`SpellCompiler.cs`（兩個 BlockType 編譯成同一個 `OpCode.Broadcast`）
- 問題：UI 和 BlockType enum 暗示有不同語意（廣播後等待回應），但實際不是。可能讓使用者困惑。
- 修法：短期在 `ScratchCanvas.BlockName` 加標注「（等同廣播）」；長期實作真正的 request/reply 語意（廣播後在 EventBus 等待一個 reply 訊號）。

**D-6｜新增積木需同步修改 `ScratchCanvas` 四個 switch** ✅
- 位置：`ScratchCanvas.cs` 的 `AddParams`、`BlockColor`、`BlockName`、`MakeDefaultBlock`
- 問題：每加一個 `BlockType`，需在四個獨立 switch 分別新增，容易漏改，且散落於 ~700 行範圍內。
- 修法：定義 `BlockDescriptor record`（color, name, defaultParams, buildParamsUI delegate），改用靜態字典 `Dictionary<BlockType, BlockDescriptor>` 集中管理，四個 switch 統一查表。

### 🟢 低優先（已全部修完 ✅）

**L-1｜`WhileIterationsTotal` 跨巢狀循環累計** ✅
- 位置：`ExecutionContext.cs`（單一計數器）、`ExecutionLoop.cs` `WhileCheck`
- 問題：`RepeatN(20)` 包 `RepeatWhile` 時，500 次上限跨所有外層迭代累計，而非單次 while 迴圈獨立計算。可能使正常設計的技能提前 Fizzle。
- 修法：改用 `Stack<int> WhileIterCounters`，`WhileCheck` push/pop 並分別計數；或將上限改為更寬鬆的值後標注文件。

**L-2｜`EntityInfo.Index` 依賴 `EnemyManager.Enemies` 列表穩定性** ✅
- 位置：`SpellCaster.cs` `QueryEnemies`（Index = 列表位置）、`SpellCaster.cs` / `SpellRunner.cs`（`enemies.Enemies[idx].TakeDamage`）
- 問題：Index 在查詢時捕捉。目前重生是原地覆蓋，安全。若未來動態插入 / 移除敵人，Index 失效指向錯誤目標。
- 修法：為 `Enemy` 加入穩定的 `int Id`（遞增），`EntityInfo` 改存 `Id`，查傷害時用 `enemies.Enemies.First(e => e.Id == id)` 查找。

**L-3｜`SetEntityProp` 只支援 "hp" 扣血** ✅
- 位置：`ExecutionLoop.cs` `StoreEntityProp`（僅處理 `prop == "hp"`）、`ScratchCanvas.cs` AddParams（下拉只有 "HP"）
- 問題：`GetEntityProp` 可讀 x/y/hp/maxhp，但 `SetEntityProp` 只能扣血，無法修改位置或其他屬性。
- 修法：`StoreEntityProp` 新增 `"x"` / `"y"` case，寫入 `PendingEntityMove`（類似 `PendingEntityDamage`），SpellCaster/Runner 消費後呼叫 `ApplyForce` 或直接設定位置。

### 問題 2｜法陣編輯器 UI 現狀分析

**現狀是否預期？** 是，屬於過渡狀態，但有兩個根本限制尚未處理：

**限制 A：積木無法拖動（Godot ScrollContainer 衝突）** ✅ **已修復（2026-06-09）**
- 根因：ScratchCanvas 的積木拖拉把手（色條）使用 `GuiInput` + `ForceDrag` 實作。但 ScratchCanvas 被包在 `ScrollContainer` 內，Godot 的 `ScrollContainer` 會優先攔截 `MouseButton` 事件，導致 `GuiInput` 裡的 `ForceDrag` 幾乎無法觸發。
- **修復方式**：改用 Godot 4 原生虛擬方法 `_GetDragData` / `_CanDropData` / `_DropData`（在 Viewport 層直接處理，不經過 ScrollContainer 事件鏈）。`DragHandle : Panel`（partial sealed）覆寫 `_GetDragData`，`DropZone : Control`（partial sealed）覆寫 `_CanDropData` / `_DropData`，刪除原有 `GuiInput` 與 `ForceDrag` 呼叫。commit: `0c20fcb`

**限制 B：與 Scratch 模式相去甚遠**
- 根因：目前的 `AbilityEditorUI` 是「舊版插槽面板 + 嵌入式 ScratchCanvas」的混合體。ScratchCanvas 被擠在畫面下半部，左側仍是圖騰庫（插槽系統），積木類型 palette 按鈕埋在 AbilityEditorUI 的某個區域，不突出。

### R-4-RENDER｜形狀接合視覺問題

> 來源：`docs/plan-mining-placement.md § R-4-RENDER`

**問題描述**：玩家先放一顆石頭球體，再放一個石頭圓錐貼在球體旁，兩者接觸邊是否有「縫隙感」？

**分析（三種案例）**：

| 案例 | 結果 | 原因 |
|------|------|------|
| 同材質、形狀連通（tile 相鄰） | ✅ 無縫 | Greedy Meshing 合併相鄰同材質面，渲染層完全無縫 |
| 同材質、形狀相切但有 1-tile 空隙 | ❌ 有空隙 | 空隙處是 Air tile，本來就該空著（設計問題，非 bug） |
| 不同材質相鄰 | 顯示材質邊界 | 符合預期（石頭 ≠ 泥土） |

**關鍵結論**：「形狀工具」只是一次放置多個 tile 的快捷鍵。一旦 tile 落地，世界不記得它是球還是錐。Greedy Meshing 處理的對象是 tile，不是形狀——**相同材質、tile 相鄰 → 自動無縫，不需要額外程式碼**。

**真正的接合問題（幾何層）**：

當玩家把一個球體（半徑 3）與圓錐底部（底面半徑 5）相接時：
- 球體的「頂點」截面只有約 1 tile 寬
- 圓錐底面卻有 5×5 tile 寬
- 結果：圓錐底面外圈的 tile 懸在球體外側，只有中心 1 tile 連接

這是**幾何設計問題**，不是渲染 bug。

**解法選項**：

| 方案 | 說明 | 狀態 |
|------|------|------|
| 玩家自主選型（預設） | 文件說明「圓弧形狀的接合面需半徑匹配」；圓柱-圓柱同半徑天然無縫 | ✅ 目前行為 |
| V 系列 Junction Fill | 放置形狀時，偵測面法線方向接觸面，自動補上接合 tile；實作為 `PlacementShape.cs` 中的 `FillJunction(world, from, to, mat)` 靜態方法 | ⬜ V 系列再加 |
| R-6 雕刻工具（Sculpt） | Push/Pull 工具讓玩家手動補坑，更靈活 | ⬜ R-6 規劃中 |

**待辦行動**：

- [ ] **V-4**（V 系列）：`PlacementShape.cs` 新增 `FillJunction(TileWorld3D world, GridPos from, GridPos to, MaterialType mat)` 靜態方法；放置時偵測接觸面法線，自動補 tile
- [ ] **R-6**（未來）：雕刻工具 Sculpt，Push/Pull 手動修形
