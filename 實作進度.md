## 程式碼待辦（Code Review 發現）

### 🔴 Bug / 正確性（已全部修完 ✅）

**B-1｜`If` 積木的 `ElseBranch` 在 UI 完全不可用** ✅
- 位置：`ScratchCanvas.cs` `BuildBlockCard`（只渲染 `ThenBranch`，未渲染 `ElseBranch`）
- 問題：`SpellCompiler` 已正確編譯 `ElseBranch` 的跳轉邏輯，但使用者無法從 UI 加入積木到 else 分支，`If` 事實上變成沒有 else 的條件。
- 修法：在 `BuildBlockCard` 的 `If` 判斷區新增 `BuildBranch(block.ElseBranch, "ELSE", indent + 1)`。

**B-2｜`SpellCompiler.EmitBlock` 靜默丟棄未知 `BlockType`** ✅
- 位置：`SpellCompiler.cs` `EmitBlock`（switch 無 `default` 分支）
- 問題：新增 `BlockType` 若忘記加 compiler case，積木在執行時無聲消失，沒有任何錯誤或警告。
- 修法：加入 `default: GD.PushWarning($"[SpellCompiler] 未處理的 BlockType: {block.Type}");`

**B-3｜`EntityInfo.Hp` 在 `ForEachNearby` 迭代中陳腐** ✅
- 位置：`ExecutionLoop.cs` `ForEachStart` / `ForEachStep`（`EntityInfo` 為施放時快照）
- 問題：`SetEntityProp` 扣血後實際 `Enemy.Hp` 更新，但 `CurrentIterEntity.Value.Hp` 和 `_e.hp` 變數仍是舊值。同次迭代呼叫 `GetEntityProp "hp"` 會讀到錯誤數值，依此計算的後續 `SetEntityProp` 也會錯。
- 修法：`SpellCaster` / `SpellRunner` 消費 `PendingEntityDamageIdx` 後，同步更新 `ctx.CurrentIterEntity` 的 Hp 欄位（需重建 record）並重寫 `_e.hp`。

**B-4｜`GlobalVars` / `GlobalLists` 從不重置** ✅
- 位置：`ExecutionContext.cs`（兩個 `static Dictionary`）；`EventBus.cs`（`static HashSet _active`）
- 問題：這三個靜態集合在遊戲重啟或場景重載時不會清除。同一 process 跑第二局，全域變數帶著上局狀態。`EventBus` 有 `ClearFrame()` 但沒有 session-level reset。
- 修法：在 `Main._Ready()` 加入 `ExecutionContext.GlobalVars.Clear(); ExecutionContext.GlobalLists.Clear(); EventBus.ClearAll();`（需在 EventBus 新增 `ClearAll` 方法）。

**B-5｜`MaxComboDepth = 5` 重複定義** ✅
- 位置：`SpellCaster.cs:11` 與 `SpellRunner.cs:23` 各有一份
- 問題：若只改一處，兩條執行路徑的連段上限悄悄走岔。
- 修法：移至 `SafetyGuard.cs` 作 `public const int MaxComboDepth = 5;`，兩處改為引用。

### 🟡 設計 / 技術債（已全部修完 ✅）

**D-1｜`SpellCaster.ExecuteEffects` 與 `SpellRunner.Advance` 高度重複** ✅
- 位置：`SpellCaster.cs`（`ExecuteEffects` 方法）、`SpellRunner.cs`（`Advance` 方法）
- 問題：兩者都做「建 ctx → 驅動 loop.Step → 消費 PendingInvokeTotem / InvokeSpell / EntityDamage → 處理 ForEach 位置覆蓋」。每次新增 pending 機制（目前三種），兩個地方都要同步修改。
- 修法：提取 `static void AdvanceContext(ExecutionContext ctx, ExecutionLoop loop, ContextRuntime rt, float delta)` 共用方法，`rt` 封裝 player / world / enemies / loadout 等依賴。

**D-2｜`ForEachNearby` 的 `player.Position` 暫時覆蓋** ✅
- 位置：`SpellCaster.cs` 與 `SpellRunner.cs` 的 `PendingInvokeTotem` 處理段
- 問題：用 save/restore 手法改寫共享物件欄位。目前 `SpellRunner` 的多個 `ActiveSpell` 是同幀序列執行，不存在競態。但若未來加入真正並行（Task / async），此處是 race condition 的埋雷點。

### 問題 2｜法陣編輯器 UI 現狀分析

**現狀是否預期？** 是，屬於過渡狀態，但有兩個根本限制尚未處理：

**限制 A：積木無法拖動（Godot ScrollContainer 衝突）** ✅ **已修復（2026-06-09）**
- 根因：ScratchCanvas 的積木拖拉把手（色條）使用 `GuiInput` + `ForceDrag` 實作。但 ScratchCanvas 被包在 `ScrollContainer` 內，Godot 的 `ScrollContainer` 會優先攔截 `MouseButton` 事件，導致 `GuiInput` 裡的 `ForceDrag` 幾乎無法觸發。
- **修復方式**：改用 Godot 4 原生虛擬方法 `_GetDragData` / `_CanDropData` / `_DropData`（在 Viewport 層直接處理，不經過 ScrollContainer 事件鏈）。`DragHandle : Panel`（partial sealed）覆寫 `_GetDragData`，`DropZone : Control`（partial sealed）覆寫 `_CanDropData` / `_DropData`，刪除原有 `GuiInput` 與 `ForceDrag` 呼叫。commit: `0c20fcb`

**限制 B：與 Scratch 模式相去甚遠**
- 根因：目前的 `AbilityEditorUI` 是「舊版插槽面板 + 嵌入式 ScratchCanvas」的混合體。ScratchCanvas 被擠在畫面下半部，左側仍是圖騰庫（插槽系統），積木類型 palette 按鈕埋在 AbilityEditorUI 的某個區域，不突出。
- 真正的 Scratch 模式需要的架構：左側改為積木類型 palette → 右側為純 canvas 工作區 → 移除插槽面板主結構。
- 這是**架構層面的改動**，非現有 Phase 3 範圍，應列為下一輪的獨立任務。
