## 1. 設計哲學

### 三個靈感來源的整合

| 來源 | 貢獻 |
|------|------|
| **Scratch** | 視覺化積木介面、函式容器、拖拉組裝 |
| **Noita** | 執行流語義、物理互動、資源作為熔斷器 |
| **Psi（Minecraft Mod）** | 節點邏輯、變數運算、實體查詢 |

```
玩家打開能力編輯器

外層：Scratch 積木介面
      拖拉圖騰、連接節點、組裝法陣
            ↓
中層：Noita 執行語義
      圖騰的觸發順序、修飾流向、時機控制
            ↓
內層：Psi 邏輯深度
      變數運算、實體查詢、條件分支
            ↓
底層：世界物理
      能力效果落地，世界真實回應
```

### 核心設計原則

- **限制即資源**：限制條件越多，開放的能力點越多
- **世界承載能力**：能力的重量感來自世界物理的真實回應
- **漸進式無敵**：早期只能操作物理層，後期能操作世界規則本身
- **代價對應強度**：最強的能力必須付出最根本的代價
- **設計與執行分層**：設計複雜法陣（創作、探索）和操作複雜能力（時機、反應）是兩個獨立的技術維度。複雜的法陣設計不應要求複雜的操作才能發揮；簡單的法陣也可以因為高超操作而超越設計值。兩個維度各自有其深度，不強迫疊加
- **資訊漸進揭示**：基礎刻印效果完全公開（設計介面即時顯示）。法則碰撞的結果效果不主動公開，先備條件達成時才顯示條件，實際使用後才顯示效果。玩家的圖鑑是自己創造的技能收藏庫，沒有「等待發現的隱藏配方」

## 3. 圖騰系統

### 圖騰法陣（類似函式）

- 包覆多個圖騰，讓圖騰間的訊號以節點相連傳輸
- 可自由命名，可被其他法陣呼叫
- 等同於 Noita 的「法杖」＋ Psi 的「節點圖」

### 圖騰名稱標識符

法陣內的每個圖騰可選擇性賦予一個名稱標識符（選填，預設為空）：

```
規則：
  名稱由設計者在介面中自行填寫，顯示於圖騰節點上方
  同一法陣內的圖騰名稱不可重複
  未命名的圖騰不可被邏輯積木指名引用

用途：
  邏輯積木可靠名稱在法陣內部觸發或條件判斷指定圖騰
  打亂模式下執行順序隨機，但指名引用不受順序影響，仍可靠

示例：
  法陣內三個圖騰分別命名「主攻」「追加」「護盾」
  邏輯積木設計：
    如果「主攻」命中成功 → 觸發法陣內圖騰「追加」
    如果「主攻」未命中   → 觸發法陣內圖騰「護盾」
```

### 六種原型圖騰

| 圖騰 | 作用 | 類比 |
|------|------|------|
| **觸發圖騰** | 廣域範圍群傷（扇形、周身、射線、FPS、以自身為中心的爆發等） | Noita 的投射物法術 |
| **武技圖騰** | 媒介施放（劍技、拳擊、弓箭等） | 近戰執行者 |
| **變幻圖騰** | 角色狀態改變（buff、變身、飛行、隱身等） | 自身修飾 |
| **領域圖騰** | 改變環境（結界、地形、天候） | 場地控制 |
| **召喚圖騰** | 召喚實體（寵獸、機械、建築、載具） | Noita 的「投射物即執行者」 |
| **位移圖騰** | 位置轉換（瞬移、衝刺、閃避） | 移動執行 |

### 圖騰合成

- 單一技能可同時觸發兩種以上圖騰效果
- 圖騰等級越高，特效越精美

### 跨法陣協同（Synergy）

> ⚠️ 後期世界觀內容，導入 MP 類型設定後才開始設計，目前僅作方向性參考。

兩個（或以上）不同法陣搭配使用時，可觸發超出個別法陣效果的「協同爆發」：

```
概念方向（參考，非規格）：
  法陣 A 命中目標 → 目標身上施加「協同標記」
  法陣 B 命中同一目標 → 讀取協同標記，觸發「Synergy 爆發效果」
  爆發效果的性質由兩個法陣的 MP 類型組合決定（對應屬性碰撞邏輯）

具體規則（協同標記時間、MP 組合對應、多法陣協同等）待世界觀設定後統一設計。
```

### 圖騰等級門檻

高階圖騰不是花點數就能解鎖，而是受角色玩家等級硬性限制：

| 圖騰類型 | 解鎖門檻 | 說明 |
|---------|---------|------|
| 觸發圖騰、武技圖騰 | 無門檻 | 初始即可使用 |
| 變幻圖騰、位移圖騰 | 玩家 LV 20+ | 基礎自身改造能力 |
| 召喚圖騰 | 玩家 LV 30+ | 需一定修為才能具現外部實體 |
| 領域圖騰 | 玩家 LV 50+ | 改變環境的高階能力 |

> 門檻未達時，圖騰在設計介面中顯示為灰色鎖定，但仍可預覽效果說明。

### 變幻圖騰——形態切換規格

變幻圖騰的「變身效果」可覆蓋以下六個維度（各項均為選填，未設定的維度保持原值）：

```
1. 數值（Stat Override）
   修改角色的任意數值屬性（HP 上限、MP 上限、攻擊力、防禦力、移速等）
   三種覆蓋模式：
     加法：變身後 HP 上限 = 原值 + 500
     乘法：變身後攻擊力 = 原值 × 2
     固定值替換：變身後移速固定為 X（無視原值）

2. 屬性（Attribute Override）
   替換角色當前持有的元素資格
   範例：原本持有火屬性 → 變身後改為暗屬性
   → 變身期間的技能與投射物均以新屬性計算元素互動
   → 變身結束後恢復原本屬性資格

3. 外觀（Appearance Override）
   替換角色的視覺模型、顏色主調、特效粒子
   由設計者選擇預設的「型態外觀資源」（外觀資源由美術製作，非設計系統範疇）

4. 名稱（Name Override）
   戰鬥介面顯示名稱替換（不影響玩家帳號名稱，僅戰鬥中顯示）
   範例：「王小明」→「混沌使者」

5. 動作組（Animation Set Override）
   切換到另一套預設的動畫集
   影響攻擊、移動、受傷、施放等所有動作的外觀呈現
   不改變技能的邏輯執行，只改變外觀

6. 技能方案（Skill Set Override）
   切換到玩家預先設定好的另一個技能方案（Loadout）
   變身期間可使用的法陣完全替換為方案 N 的配置
   變身結束後自動回到原方案
```

#### 持續與還原規則

```
持續機制：
  套用效果持續時間刻印（定時型 / 脫戰型 / 永久型均可）
  若為開關式法陣，維持時間由玩家控制（可選持續消耗 MP）

可逆性（設計者決定）：
  可逆（預設）：持續期間可主動施放「解除變身」法陣切回原型
  不可逆：持續期間無法主動中斷，必須等到到期或條件解除

巢狀變身（已處於變身狀態時再發動另一個變身）：
  替換（預設）：新形態完全覆蓋舊形態（新數值以原始值為基底計算，不疊加）
  禁止：宣告失敗（設計者可設定此選項以保護特定形態）
  疊加：兩套覆蓋同時生效（⚠️ 數值設計風險高，不建議輕易使用）

死亡處理：
  以變身狀態下的 HP 上限計算死亡
  死亡後以原型外觀重生（若有復活機制）
```

> **範例對應（附錄範例三「道終殘螢」）：**  
> 數值 → 靈力 ×3、移速 ×2、爆擊率 +30%（乘法）  
> 外觀 → 人物形象轉為型態 2  
> 技能方案 → 切換為方案 2  
> （名稱、屬性、動作組在此範例未設定，保持原值）

### 法陣命名系統

玩家自由命名每個法陣，名字是個人化的起點，也是圖鑑的主要索引。

```
命名流程：
  法陣名稱欄位預設空白，玩家必須自行填寫才可儲存。
  命名欄旁有骰子按鈕，點擊後系統跳出隨機名字建議供參考，
  玩家可採用或自行填寫，骰子建議純為靈感輔助，無特殊含義。

里程碑稱號（系統行動回饋）：
  稱號來自玩家的具體創作行動，例如：
    首次儲存法陣
    在單一法陣中使用三種以上不同類型的圖騰
    設計並儲存超過 N 個法陣
    …（具體里程碑待細化）
  稱號顯示於法陣圖鑑中，作為創作記錄的一部分。
  稱號基於玩家行為，不依賴刻印或圖騰的特定組合辨識。

記錄位置：
  玩家個人圖鑑 = 自己設計過的所有法陣的收藏庫
  法陣創建即收錄，無需額外解鎖步驟
```

### 法陣連段

法陣可以透過邏輯刻印積木「自身發動法陣『』」呼叫另一個法陣，形成連段：

```
基本流程：
  法陣 A 的圖騰執行完畢
  → 邏輯積木觸發「自身發動法陣『B』」
  → 法陣 B 開始完整執行（消耗 B 的完整 MP 代價）
    → 法陣 B 的圖騰執行完畢
    → 邏輯積木觸發「自身發動法陣『C』」
    → 法陣 C 開始完整執行...
```

**連段深度無上限**，玩家可以設計任意長度的連段鏈。

#### 執行細節

```
執行順序：串行（A 完全執行完畢後，B 才開始），非並行
MP 消耗：每個法陣消耗自身的完整 MP 代價，不因在連段中而打折
施放延遲：每個法陣的施放延遲各自獨立計算，不因在連段中而減免
觸發時機：邏輯積木在圖騰執行流末端觸發，屬於該法陣執行的最後一步
```

#### 連段的終止條件

```
以下任一情況發生時，連段終止，後續法陣不再執行：

1. MP 耗盡
   → 下一個法陣的 MP 代價超過當前剩餘 MP
   → 連段在此處自然終止，已執行的效果保留

2. 目標條件不符（若連段法陣設有觸發條件）
   → 法陣發動前判定條件失敗（無法施放，見 §15 施放失敗類型 1）
   → 該法陣 MP 不消耗，不執行，連段在此中斷

3. 邏輯積木沒有指定下一個法陣
   → 法陣執行完畢後不繼續，連段自然結束

4. 法陣在連段中被「斷招」效果命中
   → 該法陣中斷，連段在此停止
   → 此前已完成的法陣效果不回滾
   → 「斷招」與「不可打斷」均為刻印效果，
     法陣是否可被打斷由設計者透過刻印決定，非系統強制

5. 連段中的法陣在連鎖堆疊中被無效化
   → 該法陣宣告已發出（MP 已消耗），但效果不生效
   → 連段在此終止，後續法陣不再執行
   → 此前已完成結算的法陣效果不回滾
```

#### 循環連段

```
玩家可以設計 A → B → A 的循環連段（A 呼叫 B，B 再呼叫 A）。
這是合法設計，系統不以邏輯判斷阻止循環。

循環受 MP 和施放延遲控制：
  → 每輪循環消耗一次 A 的 MP 和一次 B 的 MP
  → 每個法陣的施放延遲確保循環不在單幀內無限執行
  → MP 耗盡時循環自然終止

詳見 §9「設計備註（無限連段）」。
```

#### 連段與連鎖的區別

> 連段（此處）= 玩家主動設計的**單方**順序執行鏈  
> 連鎖（§10）= **多方**能力同時觸發時的優先級裁決
>
> 連段中的每個法陣是獨立的「能力發動事件」，  
> 若為宣告型，對手仍可在其宣告窗口內連鎖反制。

### 流經型刻印（新增）

```
局部刻印：只影響下一個圖騰（Noita 修飾法術的概念）
全域刻印：影響整個圖騰法陣內所有圖騰
```

## 4. 三種執行容器

這是整個能力系統的核心擴展概念：
**不只召喚實體能裝載技能，任何獨立存在或發生的事件都可以是執行容器。**

這直接來自 Noita 的「投射物即執行者」設計——
投射物不只是造成傷害的東西，它是一個移動中的圖騰法陣。

### 三種容器對比

| 容器 | 存在方式 | 觸發時機 | Noita 對應 |
|------|---------|---------|-----------|
| **召喚實體** | 持久，有 HP，有 AI | 主動行動或條件觸發 | 念獸、寵獸 |
| **投射物** | 短暫，受物理影響 | 命中目標／地形／時間到期 | 觸發法術（Trigger）|
| **接觸** | 無獨立存在，瞬間發生 | 攻擊命中、被擊中、碰撞 | 近戰法術（Melee）|

### 屬性繼承

投射物與召喚實體繼承施法者的屬性資格：

```
施法者持有火屬性 → 其投射物視為持有火屬性
投射物命中時觸發火屬性效果：
  → 「發動者」為投射物
  → 發動資格來自施法者繼承，由施法者的屬性擁有情況決定是否可發動
```

屬性資格在施放時快照一次，之後不受施法者狀態改變影響（例如施法者在投射物飛行途中失去火屬性，投射物仍可發動）。

### 巢狀執行示意

```
角色施放技能
    ↓
飛出「投射物 A」[裝載圖騰法陣 A]
    ↓ 命中目標
執行圖騰法陣 A
    → 召喚「實體 B」[裝載圖騰法陣 B]
        → 實體 B 發動「接觸」[裝載圖騰法陣 C]
            ↓ 接觸命中
        執行圖騰法陣 C
            → 飛出「投射物 D」[裝載圖騰法陣 D]
                → ...（可無限巢狀，受執行安全機制限制）
```

### 投射物容器的參數

```
基本參數：
  數量、方向、半徑、移速、存在時間

裝載內容：
  → 一個完整的圖騰法陣

執行時機（擇一或多）：
  ├── 命中生物時
  ├── 命中地形時
  ├── 存在時間到期時
  └── 持續存在期間每 X 秒執行一次（輪詢）
```

### 接觸容器的參數

```
觸發方向：
  ├── 主動接觸（我攻擊別人時）
  ├── 被動接觸（被別人打到時）
  ├── 雙向接觸（互相碰撞）
  └── 移動穿越（路過某生物時）

裝載內容：
  → 一個完整的圖騰法陣

執行對象：
  ├── 對接觸目標
  ├── 對自身
  └── 對雙方
```

#### 持續接觸的生命週期鉤子

當接觸容器設計為「持續掛載在目標上」時（類似 Buff），具備三個生命週期鉤子，  
各自可裝載獨立的圖騰法陣：

```
掛載時（OnAttach）：
  → 接觸效果首次命中目標時觸發一次
  → 適合：初始傷害、施加標記、播放特效

週期時（OnTick）：
  → 持續掛載期間每 X 秒觸發一次
  → 適合：持續 DoT、持續減速、持續能量吸取

移除時（OnRemove）：
  → 效果到期或被主動解除時觸發一次
  → 適合：解除後爆炸、恢復被改變的狀態、觸發連鎖效果

設計意圖：
  三個鉤子讓持續型接觸效果有清晰的「開始 / 持續 / 結束」結構，
  玩家可以只填其中一個或多個（未填的鉤子不執行）。
  靈感來自 Warcraft 3 的 Buff 生命週期模型。
```

### 容器觸發效率係數（Proc Coefficient）

```
每個容器類型有預設的「觸發效率係數」（0.0–1.0），
決定 on-hit 刻印（命中觸發效果）被引發的機率倍率：

  玩家本體直接觸發：1.0（完整機率）
  召喚實體攻擊：    0.7
  投射物（追蹤型）：0.5（防止高頻追蹤投射物無限觸發 on-hit 刻印）
  接觸型（持續貼身）：0.2 per tick（防止持續接觸高頻觸發）

係數可透過設計時的刻印進一步調整（在容器設定中覆蓋預設值）。
邏輯刻印的 on-hit 觸發分支受係數影響：機率型積木乘以係數後才判斷是否觸發。
```

### 此機制解決的設計問題

**必中飛彈＋黑百足蟲（HxH）**：
```
投射物「飛彈」
  裝載法陣：命中時植入「百足蟲」實體
「百足蟲」實體
  裝載法陣：偵測到目標反抗行為 → 自身增大
```

**接觸型詛咒**：
```
接觸觸發（主動）
  裝載法陣：植入追蹤標記 + 傷害計算
→ 每次攻擊命中時自動執行，不需額外觸發條件
```

**遞迴爆炸型能力**：
```
投射物 A 命中 → 生成 N 個投射物 B
投射物 B 命中 → 生成 N 個投射物 C
（受執行安全機制的實體數量上限控制，不會真正無限）
```

## 5. 符文與能量系統

### 角色能量符文

- 共 12 種 MP 類型，與角色種族或訓練有關
- 能量欄位同時承受上限：預設 3 種，特殊方式可達 5 種
- 超過上限發生衝突自爆
- 多餘能量儲入「儲能瓶」，可隨時切換使用

> **實作說明：** 目前所有 MP 類型統一以「魔力」代稱實作（因此所有技能的 MP 優先級暫時相同）。
> 系統架構必須預留 MP 類型切換介面，待世界觀完成後可直接替換為對應種類，無需重寫邏輯。

> **重要概念區分：** MP 類型（能量符文，12 種）與屬性刻印（元素，11 種）是**完全不同的概念**，不可混用。
> MP 類型決定「用哪種能量施放技能」；屬性刻印決定「技能攜帶哪種元素效果」。

### 能量消耗設計原則

- 不同 MP 類型在不同效果的損耗不同
- 能量是執行安全的天然熔斷器（耗盡即停止）

| 紅 | 侵略效果刻印 | 瞬殺機率、剝奪感官、負面效果疊加、斷招；**替代效果（負向）** |

### 刻印縮放係數

每個效果刻印有個別縮放係數，決定「投入能力點後效果提升的幅度」：
- **傷害 / 概率類刻印**：縮放係數即雙曲線公式中的 **a 值**（`f(x) = 1 − 1/(1 + a·x)`）。a 越大，曲線越陡、初期高報酬但快速飽和；a 越小，曲線越平緩、邊際報酬持續但低。
- **輔助類刻印**：縮放係數即線性公式中的 **k 值**（`f(x) = base + x · k`）。k 決定每點投入的等量增益。

以下為各係數的設計意圖參考：

| 縮放係數 | 典型刻印類型 | 設計意圖 |
|---------|------------|---------|
| 0.2 | 持續型 Buff | 曲線極平緩（a=0.2），10 點投入僅達約 67% 效果；基礎效果強，額外投入邊際報酬低 |
| 0.5 | 範圍型傷害 | 中等曲線，穩定低風險，適合多用途法陣 |
| 1.0 | 單體傷害 | 標準曲線（a=1.0），前幾點效益顯著；亦是輔助類刻印 k 的標準基準值 |
| 2.5 | 核爆型傷害 | 曲線極陡（a=2.5），1 點即達約 71% 效果；初期高回報，但基礎消耗極高才能投入此類刻印 |

縮放係數顯示於刻印說明欄，玩家設計時一目了然。

### 刻印點數投入縮放曲線

不同刻印類別在「投入更多能力點」時，應採用不同的數學曲線，
製造不同的「第 N 點的邊際效益」感受：

```
傷害類刻印（及概率觸發類）→ 雙曲線縮放（Hyperbolic）：
  效果增長越來越慢，自然趨近上限，防止無限疊加破壞平衡
  公式參考：f(x) = 1 − 1 / (1 + a·x)
  → 第 1 點效益顯著，第 100 點效益極低，但永遠有效益，沒有硬牆感

輔助類刻印（減速時長、控制時長、護盾值等）→ 線性縮放：
  每點投入效益相同，疊加感清晰，玩家易於計算
  公式參考：f(x) = base + x · k

概率類刻印（觸發機率）→ 強制雙曲線：
  防止觸發率趨近 100%，機率類效果設計上限（如最高 80%）

具體數值（a 值、k 值、上限）待 §8 數值調整時統一設定。⚠️ 數值待調整
```

### 替代效果刻印（紅色／綠色子類型）

```
格式：「若 [X 事件] 將發生 → 改為 [Y 事件]」，原事件完全取消，不進入連鎖堆疊
與其他刻印的差異：
  傷害刻印 = 修改數值（傷害 100 → 80）
  無效化刻印 = 阻止能力宣告進入宣告窗口
  替代效果刻印 = 整個事件被替換（「受傷」事件替換為「回復」事件）

紅色子類（負向替代）：
  「若目標會獲得 Buff → 改為該 Buff 轉移至施法者」（效果劫奪）
  「若此法陣命中 → 改為對命中者的盟友釋放同等效果」（目標替代）

綠色子類（正向替代）：
  「若施法者會受到致命傷害 → 改為 HP 剩 1」（死亡替代）
  「若施法者會受到傷害 → 改為回復同等 HP」（傷害轉治療）

替代效果的關鍵特性：
  → 不進入堆疊，即時生效，無法被連鎖回應
  → 多個替代效果同時適用時，受影響方選擇套用哪一個
  → 靈感來自 MTG 的 Replacement Effects
```

### 效果持續時間分類

所有效果刻印的「持續時間屬性」明確分四種：

```
即時型（Instant）：施放後立即結算一次，不存留
  → 對應積木：無計時器，執行後效果直接進入結算層

定時型（Duration）：持續 N 秒後自動移除
  → 對應積木：「持續 X 秒」計時器
  → 移除時觸發「效果結束事件」（可被邏輯積木偵測）

脫戰型（Combat-Scoped）：脫離戰鬥後緩慢衰減至 0，與積累類刻印採用相同規則
  → 對應積木：「持續至脫戰」
  → 脫離戰鬥定義：與任何敵方無傷害 / 接觸超過 X 秒（與積累類刻印採用相同定義）
  → 戰鬥中可反覆疊加；脫戰後不立即消失，而是逐漸衰減（短暫脫戰後重新接敵，效果有殘留）
  → 衰減速率：各效果可獨立設定，預設值與積累類刻印相同（⚠️ 數值待調整）
  → 適合：「每次擊殺暫時提升攻速」、「連擊累積傷害加成」等戰鬥節奏型增益

永久型（Infinite）：直到被特定事件主動移除
  → 對應積木：無計時器的被動條件
  → 解除條件由設計者明確定義（死亡 / 條件失效 / 主動取消）
  → 持續生效型能力預設使用此類型
  → ⚠️ RPG 語境警告：永久型效果跨戰鬥保留，若疊加來源無上限（如「每次擊殺永久+X」），
       會在開放世界中無限累積。此類設計在 MMORPG 中需要明確的全局上限或世界觀理由，
       與 MOBA（遊戲結束即重置）的設計邏輯根本不同。
       → 若設計意圖是「戰鬥中累積但戰後清除」，請改用脫戰型。
       → 若設計意圖是真正的永久成長（類似 Nasus），列入未來方向 §18 討論，
         待世界觀與平衡框架確立後才設計。
```

### 屬性修改分層計算

```
所有屬性修改器按固定順序計算，無論加入順序，最終值一致：

  Final Value = (Base + ΣAdditive) × ΣMultiplicative

  1. Base（基礎值）：能力點投入決定的基礎效果值
  2. ΣAdditive（加法修改器總和）：所有「+X」類型的刻印效果
  3. ΣMultiplicative（乘法修改器總和）：所有「×Y」類型的刻印效果

  設計介面顯示三欄：Base / +Add / ×Mul，讓玩家清楚看到每層的貢獻

衝突處理：
  同一屬性有多個「設為固定值」修改器 → 取最後生效的（依時間戳記）
  加法與乘法不衝突，可同時疊加

靈感來自 Unreal GAS 的 Attribute Modifier Stack。
```

### 刻印屬性層正交設計原則

```
刻印應設計為「修改法陣屬性結構的不同欄位」，而非「觸發獨立效果」。

法陣屬性欄位示例：
  投射軌跡欄位（直線 / 追蹤 / 反彈 / 繞行 orbit）
  命中效果欄位（燃燒 / 冰凍 / 感電 / 中毒）
  傷害數值欄位（基礎傷害 × 縮放係數）
  觸發條件欄位（即時 / 條件觸發 / 開關式）
  範圍欄位（AoE 半徑 / 穿透 / 分裂數量）

不同欄位的刻印可自由疊加，系統自動合成結果：
  追蹤刻印（軌跡欄位）+ 冰凍刻印（命中效果欄位）
  → 自動合成「追蹤投射物命中附加冰凍」，無需手動硬編碼這個組合

相同欄位的刻印發生衝突：
  系統依刻印優先序取一（高優先級者生效）
  設計介面標出衝突警告，提醒玩家有刻印被覆蓋

設計重點：越多效果透過欄位合併運算，湧現組合空間越大；
          越多效果寫成「遇到 X 才做 Y」，組合空間越死板。
          （靈感來自 Binding of Isaac 的淚滴屬性層設計）
```

### 積累類刻印規格

適用於所有「積累層數 → 閾值觸發 → 爆發效果」機制的刻印
（包含現有的層疊刻印、引爆刻印，以及屬性碰撞產生的狀態效果）：

```
積累衰減規則：
  脫離戰鬥（對目標無任何傷害/接觸超過 X 秒）後，積累量緩慢衰減至 0
  ⚠️ 衰減速率與緩衝時間待數值調整

多源積累疊加規則：
  同種積累效果由多個法陣同時施加時，積累量直接相加（不取最大值）
  例：法陣 A 每次攻擊施加 30 層燃燒積累，法陣 B 施加 20 層
  → 同一攻擊命中時對目標施加 50 層燃燒積累

閾值觸發後歸零規則：
  積累量達到閾值觸發爆發後，該目標的此種積累量立即歸零
  重新從 0 開始積累（防止秒連觸發）
  → 玩家需再次疊加到閾值才能再次觸發
```

### 斷招機制細化（Phase 2）

```
現有：「斷招刻印」= 直接打斷目標正在執行的法陣（即時二元判斷）

Phase 2 細化方向（姿態積累閾值打斷）：
  a. 攻擊對目標造成「姿態傷害」，不同攻擊類型係數不同：
       大範圍慢速攻擊：高姿態傷害（容易打斷但難命中）
       多段快速攻擊：低姿態傷害（單次不易打斷，但密集累積有效）
       魔法攻擊：中等姿態傷害

  b. 姿態傷害累計超過目標的「姿態耐久閾值」→ 觸發強制打斷
       被打斷後有短暫的「姿態無敵幀」（防止無限連鎖打斷）
       姿態耐久在一段時間後自動回復（閾值重置）

  c. 「不可打斷刻印」（藍色圖騰改造刻印）：
       現有版本：完全免疫打斷
       Phase 2 版本：提升姿態耐久閾值（不再完全免疫，但大幅提高）

此機制在 Phase 2 替換現有簡單版本，現階段維持直接打斷設計。
```

### 軌跡操控刻印（配對使用）

此對刻印讓投射物容器的飛行路徑可以被記錄並反向重播，實現「回旋鏢」、「飛彈原路飛回」等效果。兩個刻印需配對使用。

```
軌跡記錄刻印（藍色圖騰改造刻印）：
  套用於投射物容器
  投射物存在期間，每 tick 記錄當前位置，建立軌跡歷史序列
  最多記錄 N 秒的歷史（N 由能力點決定，⚠️ 數值待調整）
  → 超出記錄上限後，最舊的位置點自動捨棄（滾動記錄）
  → 投射物消失時，軌跡歷史一併清除

軌跡回播刻印（效果刻印）：
  目標：指定一個「帶有軌跡記錄」的存活實體
  效果：該實體以相同速度沿記錄路徑的反向移動
  → 回程過程中正常觸發 OnHit 等命中事件（可以打人）
  → 回到起點後停止，或觸發回播結束事件（可接法陣）
  → 若目標實體在回播中被摧毀，回播中止

關鍵差異（與時間回朔 Eidos Reversal 的對比）：
  Eidos Reversal：目標瞬間移至過去的位置（瞬移，中間過程不存在）
  軌跡回播：目標沿來路真實飛行回去（有物理移動過程，途中可命中目標）
```

> 靈感來自薩爾達傳說 TotK 的「Recall」能力。

### 限制刻印折扣透明化

黃色限制刻印在設計介面中強制顯示完整取捨數字：

```
「此刻印使能力點消耗 −X 點，效果強度削弱 Y%」
```

- 折扣比例為固定值，不受其他刻印影響
- 介面同時顯示「加入前 / 加入後」的消耗對比
- 玩家清楚知道在交換什麼，不��試錯猜測

### 代價類型標記（費用 vs 效果）

刻印中的「代價」分兩種語義，設計介面強制標記，決定替代效果刻印能否攔截：

```
費用（Cost）：必然發生，不受替代效果刻印影響
  範例：「消耗自身 X 點 HP 才能施放」、「棄置一個效果刻印後才觸發」
  → 即使對手（或自身）持有防護型替代效果，費用仍照常執行
  → 介面標記：[費用]

效果（Effect）：屬於技能結算的一部分，可被替代效果攔截
  範例：「命中後自身受到反傷 X 點」、「施放後移除自身一個 Buff」
  → 若有對應的替代效果刻印，此代價可被替換（如「反傷→回血」）
  → 介面標記：[效果]

設計介面：
  添加限制刻印時，設計者必須選擇此代價屬於費用或效果（不可留空）
  選擇後以顏色標籤顯示於刻印說明旁，對戰時雙方均可看見

設計意圖：
  確保高強度限制刻印（如自傷代價）的真實性——
  不能靠替代效果刻印「把費用也繞掉」，讓代價淪為擺設。
```

### 額外操作刻印——選擇型（Discover）

紫色額外操作刻印的子類型，讓施放者在執行時做出一次互動決策。

```
設計時：
  定義 2–4 個候選效果（各自是一組完整的效果積木）
  為每個候選效果填寫顯示文字（讓施法者在選擇時能讀懂選項意義）

執行時：
  技能執行暫停，向施法者顯示選項卡
  施法者選擇一項後，執行對應效果，繼續後續流程
  可設定選擇時間上限（⚠️ 數值待調整）；超時自動選第一個選項
  可設定「選項是否對對手可見」（透明或隱藏）

能力點消耗：
  此刻印的能力點消耗 = 所有候選效果能力點之和（等同設計了 N 套備用效果）
  選項越多，消耗越高；每次施放只執行其中一個，但點數全額計入

靈感來自爐石戰記的 Discover 機制。
```

### 基本積木（Scratch 概念）

```
判定積木：
  如果「」
  重複執行「」

  選項 1：「效果積木組」
```

### 資料結構積木

現有持久變數積木僅能儲存單一值。當設計需要「有序多元素集合」時（如武器輪換、
配方組合、技能使用歷史），需要以下列表積木。

```
── 列表積木（List）─────────────────────────────
  建立列表「」= [A, B, C, ...]   ← 宣告具名有序列表（可選填初始內容）
  列表「」的長度                 ← 當前元素數量（→ 數值）
  列表「」第「」項               ← 按 index 讀取（1-based；越界回傳空值）
  設定列表「」第「」項 = 「」     ← 修改指定位置的值（不消耗元素）
  將「」加入列表「」末尾          ← Append（佇列的 Enqueue）
  取出列表「」頭部元素            ← 消耗性讀取，元素移除（佇列的 Dequeue）
  取出列表「」尾部元素            ← 消耗性讀取，元素移除（堆疊的 Pop）
  從列表「」移除第「」項          ← 按 index 移除
  清空列表「」
  列表「」是否包含「」            ← 成員查詢（→ 布林）

組合用法：
  佇列（FIFO）：加入末尾 + 取出頭部 → 輪換型資源（如武器佇列）
  堆疊（LIFO）：加入末尾 + 取出尾部 → 後進先出
  環形緩衝   ：取出頭部後立刻加入末尾 → 元素循環不消失
  集合用途   ：「是否包含」+ 只在不含時才加入 → 追蹤「本局已使用的技能種類」
```

> **注意：** 列表屬於法陣內部的資料結構。若需跨法陣共享，宣告為全域（與全域持久變數相同機制）。  
> 列表長度無硬性上限，但元素過多時佔用計算預算，影響能力點消耗。

### 被動反應觸發積木（自動偵測類）

此類積木不需玩家主動施放，由系統自動偵測條件後觸發，  
是設計被動防禦、自動反應、條件反射型能力的核心工具。

```
── 投射物偵測 ──────────────────────────────
  偵測到投射物進入「」範圍內（單位：米）
  偵測到投射物從「前方/後方/側面/任意方向」接近
  偵測到投射物屬性類型為「物理/火/冰/雷/毒/...」
  偵測到投射物來源為「敵方/友方/自身」

── 攻擊動作偵測 ────────────────────────────
  偵測到「敵方/友方」正在蓄力（出手前）
  偵測到「敵方/友方」施放「近戰/投射物/AoE/召喚/持續」型能力
  偵測到「敵方/友方」使用「MP類型」的能力
  偵測到攻擊動作命中前「」毫秒（極窄窗口，高能力點消耗）

── 被擊中偵測 ──────────────────────────────
  偵測到受到「近戰/投射物/AoE/接觸/環境」傷害
  偵測到單次受到超過「」點傷害
  偵測到受到「狀態類型」效果施加（如：被燃燒、被凍結、被標記）

── 自身狀態閾值偵測 ────────────────────────
  偵測到自身 HP 低於「」%
  偵測到自身 MP 低於「」%
  偵測到自身進入「狀態類型」（獲得/失去）
  偵測到自身連擊數達到「」次

── 目標/環境偵測 ────────────────────────────
  偵測到目標 HP 低於「」%
  偵測到「敵方/友方/物件」進入「」範圍內
  偵測到「特定地形/材質」在腳下/前方「」格內
```

> **設計注意**：偵測窗口越精確（如「命中前 X 毫秒」），能力點消耗越高。  
> 連續偵測型積木（持續監聽）比一次性偵測消耗更多，  
> 防止玩家用零成本的持續偵測積木製造無限反應鏈。

### 觸發時機模式

被動反應觸發積木預設在「條件滿足的當下」立即觸發。可附加觸發時機模式修改此行為：

```
任意時機（預設）：
  條件滿足即觸發，不等待此 tick 後續事件

末端時機（End-of-Chain）：
  條件滿足後暫緩觸發，等到本 tick 結算鏈完全清空後才執行
  → 若本 tick 仍有後續事件在堆疊中，繼續等待
  → 所有事件結算完畢後，才以「收尾效果」的形式觸發

  適合：
    「每次連鎖結算結束後，回復 X HP」
    「所有本 tick 傷害結算完畢後，若目標仍存活則觸發追加」
    「本 tick 最後一個事件結算後，移除施法者身上所有 Debuff」

  代價：末端時機型被動能力點消耗高於任意時機
       （需要系統持續追蹤「此 tick 結算鏈是否已清空」）

  靈感來自遊戲王「選擇性觸發效果的時機規則」：
  When 型效果若不是一系列動作的最後事件，則錯過時機無法發動。
  本系統將此設計為可選的觸發精度，而非隱性規則。
```

### 技能操控積木

此類積木允許施法者臨時取得目標的法陣使用權，是邏輯刻印中能力點消耗最高的積木類別之一。

```
技能複製積木：
  複製目標「最後宣告的法陣」，取得一份臨時副本
  （原主人仍保有原法陣，不受影響）
  持續「」秒                      ← 必填；到期前可反覆使用；預設上限 30 秒
  以自身屬性結算                  ← 副本的 MP 消耗、傷害係數、效果強度均以施法者計算

技能掠奪積木：
  掠奪目標「最後宣告的法陣」，強制取走使用權
  （原主人在掠奪持續期間無法使用此法陣，宣告直接失敗）
  持續「」秒                      ← 必填；到期後自動歸還原主人；預設上限 15 秒
  以自身屬性結算                  ← 與複製積木相同

兩者的差異：
  複製：共享，原主人不受限制，持續時間上限較長（30 秒）
  掠奪：獨佔，原主人暫時失去法陣，持續時間上限較短（15 秒）

共同規則：
  → 觸發條件：需先命中目標，命中時才讀取目標的「最後宣告法陣」
  → 即時型法陣不可被複製或掠奪
    （即時型無宣告窗口，無法被觀察；宣告型、持續生效型可被複製/掠奪）
  → 若目標從未宣告任何法陣：Fizzle（MP 已消耗，無結果）
  → MP 代價：獲得的法陣以原法陣設計的 MP 代價為準，由施法者支付
             若施法者 MP 不足：法陣在戰鬥介面標記「MP 不足」，無法宣告
  → 副本與掠奪所得法陣不可被再次複製或掠奪（禁止鏈式複製）
  → 原主人死亡時：被掠奪的法陣隨之消失（不歸還）；副本不受影響
  → 持續時間內可反覆使用（受法陣自身冷卻限制）
  → 戰鬥介面：獲得的法陣以「臨時格」顯示，圖示與原法陣相同但帶倒數計時條
  → 能力點消耗：此類積木極高（等同設計了一個可動態替換的法陣欄位）
```

### 執行流設計（Noita 概念）

- 圖騰法陣內，刻印按順序執行
- 前置刻印影響後續圖騰（局部修飾）
- 全域刻印掛在法陣層級，影響所有圖騰

## 9. 抽象能力的實作層次

能力按照「操作對象」分為五個層次，層次越高需要越多能力點與法則刻印：

```
第一層：物理層
  操作：速度、位置、力
  範例：接近時持續減速（空間速度遞減能力）
  實作：物理引擎修飾器

第二層：狀態層
  操作：HP、buff、debuff、實體屬性
  範例：高利貸念能力、飛彈百足蟲
  實作：遊戲狀態機

第三層：行動結算層
  操作：行動是否生效、效果歸零
  範例：Gold Experience Requiem（一切歸零）
  實作：行動攔截鉤子，展示層與結算層分離

第四層：時間軸層
  操作：預知未來、選擇分支、覆蓋現實
  範例：The Almighty（猶哈巴赫）
  實作：狀態快照系統 + 模擬分支 + 狀態回滾

  多人設計原則：
    時間系效果具有明確的「影響範圍」，不作用於全局世界。
    無論暫停、回朔或預知，效果只影響指定區域內的物理引擎
    以及區域內玩家或實體的狀態。

    回朔的處理規則：
      → 回朔至「目標時間點」後，
         所有在「目標時間點之後」發動的能力一律取消
      → 被取消能力的 MP 代價、冷卻時間一併退還，
         視為該能力從未發動過
      → 效果展示動畫可播放（告知玩家發生了什麼），
         但結算層完全回滾
      → 已在區域外生效的效果不受影響
      → ⚠️ 待調整：若連鎖堆疊已結算的效果（§11.2 LIFO 結算完畢）
         遭回朔覆蓋，暫定「時間回朔」優先，已結算效果一併取消。
         此判定影響多人對戰深層互動，需在多人系統設計完成後統一確認。

第五層：定義層
  操作：改變實體「是什麼」
  範例：一文字＋刻下新名（兵主部一兵衛）
  實作：實體名字屬性映射表，覆蓋屬性定義
```

> 第一二層用屬性刻印可達；第三四五層需要對應的法則刻印，
> 且能力點消耗極高，代價刻印必須同等嚴苛。

## 9-A. 世界時間系統

> 時間是世界狀態的基礎屬性之一，需要精確追蹤，並且可以被高階能力影響。  
> 細節（月份天數、年份長度、遊戲秒換算）待世界觀設計階段補充。

### 時間單位層次

| 單位 | 換算 | 現實時間對應 | 說明 |
|------|------|------------|------|
| **Tick（幀）** | 1 tick | 50ms | 最小計時單位，系統內部時間基準 |
| **遊戲秒** | ⚠️ 待定 ticks | 待定 | 世界內時間顯示單位（如日夜周期） |
| **遊戲分** | 60 遊戲秒 | 待定 | |
| **遊戲時** | 60 遊戲分 = 1,200 ticks | 1 現實分鐘 | |
| **遊戲日** | 24 遊戲時 = **28,800 ticks** | **24 現實分鐘** | 暫定，鎖定 |
| **遊戲月** | ⚠️ 待定 遊戲日 | 待定 | |
| **遊戲年** | ⚠️ 待定 遊戲月 | 待定 | |

> ⚠️ 遊戲秒的 tick 換算尚未確定（世界時間顯示用，待細化）。  
> 目前唯一鎖定的換算：**1 遊戲日 = 28,800 ticks = 24 現實分鐘**。  
>
> **能力效果持續時間使用現實秒（1 現實秒 = 20 ticks），與遊戲世界時間系統無關。**  
> 玩家在設計介面中可調整的效果持續時間最小單位為 **tick（50ms）**。

### 時間流速與流向

時間流速是世界狀態中的一個可修改屬性（Time Scale）：

```
正常流速：Time Scale = 1.0（預設，每現實 50ms 推進 1 tick）
加速：     Time Scale > 1.0（如 2.0 = 每 50ms 推進 2 tick，世界加速）
減速：     0 < Time Scale < 1.0（世界減速）
暫停：     Time Scale = 0.0（時間靜止，物理引擎停止推進）
逆流：     Time Scale < 0.0（時間倒退，對應 §9 第四層回朔）
```

影響範圍三級：

```
全域（整個世界）：需最高法則層次的能力（根源/極限法則）
區域（指定半徑內）：§9 第四層時間軸層效果的標準範圍，不影響區域外
單體（單一實體）：中等法則層次可達，只影響目標的狀態推進
```

### 與能力系統的交叉點

```
§9 第四層（時間軸層）的能力，操作的正是此系統的 Time Scale：
  暫停 = 將目標區域 Time Scale 設為 0.0
  回朔 = 將 Time Scale 設為負值 + 回滾至狀態快照
  預知 = 讀取模擬分支（不修改實際 Time Scale）

§10 施放延遲、持續效果計時 均以 tick 計數，
受 Time Scale 影響的實體，其計時器推進速度也隨之縮放。

例：目標被「時間減速」效果覆蓋（Time Scale 0.5）時，
    施加於目標的 10 秒 Buff 實際消耗 20 現實秒才到期。
```

## 9-B. 戰鬥狀態系統

每個實體（玩家、生物、NPC）獨立追蹤自身的戰鬥狀態，
這是「本場戰鬥統計」、「脫戰型效果衰減」等機制的共同基礎。

### 戰鬥狀態定義

```
兩種狀態：
  非戰鬥狀態（Out of Combat）：預設狀態，無主動敵對互動
  戰鬥狀態  （In Combat）    ：正在進行敵對互動

進入戰鬥：
  → 主動對敵方實體施放技能 / 造成傷害
  → 被任何敵方實體攻擊 / 受到來自敵方的傷害
  → 對敵方施加持續效果（如中毒、燃燒標記）
  （上述任一條件成立即觸發進入戰鬥）

脫離戰鬥：
  → 連續 X 秒內無任何攻擊 / 傷害 / 接觸事件（對任何敵方）
  → X 的預設值與積累類刻印的脫戰衰減緩衝時間相同（⚠️ 數值待調整）
  → 脫離後再度進入戰鬥 = 新場戰鬥，本場計數全部重置
```

### 場戰鬥 ID（Battle ID）

```
每次進入戰鬥時，系統為本次戰鬥分配一個唯一 ID。
所有「本場戰鬥作用域」的統計計數器，均以此 ID 為索引。
脫離戰鬥後再次進入，分配新 ID，舊計數器捨棄。

意義：
  「本場戰鬥施放技能 N 次」= 在同一個 Battle ID 下的累計次數
  脫戰逃跑再回來 → 新 Battle ID → 計數從 0 重新開始
```

### 統計作用域

「全局戰鬥統計積木」（§7）查詢計數時，使用以下三種作用域之一：

```
本場戰鬥（Battle-scoped）：
  從當前 Battle ID 開始至今的累計
  脫戰後重新進入則重置
  適合：Storm 式累積、成長型效果、場戰鬥里程碑觸發

時間窗口（Time-window）：
  「最近 N 秒內」的滾動統計（N 由設計者在積木中設定）
  不依賴戰鬥狀態，始終滾動計算
  適合：Combo 式「剛才做了什麼」、短期行為偵測

本次連段（Combo-scoped，選配）：
  從連擊計數器開始 > 0 到歸零的期間
  連擊中斷即重置
  適合：連段多樣性統計、連段長度獎勵
```

> **設計說明**：本系統不引入「本回合（This Turn）」的概念。  
> 即時制動作 RPG 沒有自然的回合邊界；MTG 的「this turn」在功能上等同於  
> 「本場戰鬥」（長期）或「時間窗口」（短期）的組合，以上兩種作用域已完整覆蓋其設計空間。

## 10. 執行安全機制

```
問題：邏輯刻印允許「重複執行」，玩家可能寫出無限迴圈，
      或透過法陣連段形成跨法陣的無限執行鏈。

解法（不偵測迴圈，而是限制執行資源）：

1. 能量（MP）作為天然熔斷器

   MP 計算規則：
   → 部分積木與刻印本身帶有 MP 消耗值，並非所有積木都有
   → 技能設計完成後，系統統計所有積木與刻印的消耗總和，
     顯示為該技能的「發動 MP 代價」
   → 發動時一次性扣除完整 MP 代價（宣告計費包含所有積木的執行費，
     無額外的逐步扣款）
   → 介面顯示消耗即為實際消耗，不存在「隱藏的執行消耗」

   連段技能的特殊情況：
   → 連段中每個法陣各自消耗自身的完整 MP 代價
   → 技能介面顯示完整估算 MP 代價（起始法陣 + 所有可觸發法陣之和）；
     若估算總代價超過當前 MP 上限，數字以紅色顯示；
     若連段為無限循環設計（依賴 MP 回復持續施放），
     以藍色標記「直至力量枯竭」取代固定數字
   → 次輪法陣代價超過當前剩餘 MP 時，連段自然終止

2. 同時存在的效果實體數量上限
   → 新效果取代最舊的效果
   → 防止實體爆炸

3. 施放延遲
   → 圖騰有最短施放間隔
   → 迴圈變成「每 X 毫秒執行一次」而非「單幀無限執行」
   → 法陣連段中每個法陣的施放延遲各自獨立計算

4. 觸發鏈 Proc Mask
   → 每次法陣觸發鏈建立時維護一個「已觸發 on-hit 刻印記錄」
   → 同一個 on-hit 刻印在同一觸發鏈中只能啟動一次
   → 防止「刻印 A 命中觸發刻印 B → B 命中再觸發 A → A 再觸發 B...」的無限交叉迴圈
   → 允許不同刻印在鏈中各自觸發一次（A→B 合法，但 A→B→A 被截斷）
   → Proc Mask 在此次觸發鏈結束後自動重置，不影響下一次獨立觸發

5. 每 tick 執行次數上限（N = 50）
   → 不論何種機制（含快速取消刻印繞過施放延遲、高 MP 高頻連段等），
     同一 server tick（50ms）內所有法陣的執行總次數不超過 50 次
   → 超過上限後的執行延到下一個 tick 繼續（非拋棄，不影響最終效果）
   → 防止任何設計組合在單 tick 內觸發大量執行導致遊戲崩潰
   → 合理創意連段（含 RC 激進連段）通常在 20 次以內，不會觸及上限
   → ⚠️ N = 50 為初始估算，待 VM 效能測試後依實際執行成本調整
   → 來源：快速取消刻印（RC）移除施放延遲後，MP 耗盡前的單 tick 執行數量需要此上限兜底

注意：連段鏈的各法陣獨立受上述機制約束，
      不存在跨法陣的累計熔斷（每個法陣各自消耗 MP 和施放延遲）。
      若玩家使用快速取消刻印繞過施放延遲，每幀執行次數上限（機制 5）接替成為最終保障。

設計備註（無限連段）：
  若玩家設計出 MP 中性甚至 MP 正的連段鏈（回復速度 ≥ 消耗速度），
  理論上連段可無限延續。
  每幀執行次數上限確保遊戲不崩潰，
  這類設計應被允許——它是系統刻意保留的創作空間。
  玩家需要自行承擔後果：
    → 無限連段若對戰鬥無實質幫助，只是在消耗時間
    → 若能造成持續傷害或效果，等同於高成本的長期 DoT 設計
  不應以規則強制限制，而應讓 MP 設計和冷卻設計自然收束。

⚠️ 設計警惕：「快速取消刻印 + MP 正收益連段」在數學上可能形成極高頻率的持續輸出鏈。
  遊戲不會崩潰（每幀上限兜底），但每秒最多 1000 次執行的傷害潛力需在
  後期數值調整階段針對相關刻印的疊加效果特別留意。
```

### 被動狀態清理鉤子（State-Based Actions）

VM 在每個 tick 的「優先權分配前」自動執行一輪狀態基礎檢查，所有條件同時結算，不進入連鎖堆疊：

```
HP ≤ 0          → 觸發「死亡事件」
積累層數達閾值   → 觸發「爆發事件」（確保不被其他事件跳過）
持續效果時間到期 → 自動移除並觸發「效果結束事件」
Buff 條件失效   → 自動移除依附的規則效果
標籤條件不再成立 → 移除該標籤並解除對應的封鎖/授予效果

SBA 特性：
  → 同時結算，不依序
  → 不進入連鎖堆疊，無法被連鎖回應
  → 確保遊戲狀態永遠自洽（不存在「HP = -5 但仍存活」的中間狀態）

靈感來自 MTG 的 State-Based Actions。
```

## 11. 能力衝突結算系統

當多個能力同時觸發或互相干涉時，依以下三層順序處理。

---

### 11.1 第一層：優先級比較

每個能力在設計時就具有一個「優先級」，由兩個要素決定：

#### 優先級來源一：法則等級

```
根源法則（創造、次元、世界、混沌）  → 最高優先級
極限法則（陰陽、極點、無量）        → 次高
天道法則（輪迴、生死、靈魂）        → 第三
基軸法則（時間、空間、造化、因果）  → 第四
法則碰撞合成（如「時空」、「永恆」）→ 高於組成法則，低於更高階法則
屬性複合（如「金＋雷＝超載」）      → 高於單一屬性
單一屬性刻印（金木水火土冰風光暗雷毒）→ 基礎層
純物理 / 無刻印                     → 最低優先級
```

#### 優先級來源二：能量符文類型（12種 MP）

```
不同 MP 類型對不同目標有優先級加成
（具體排序依世界觀設定，待角色設計時定義）

同 MP 類型的能力互相衝突時，回歸法則等級比較
```

#### 優先級結算規則

```
高優先級能力 vs 低優先級能力：
→ 高優先級直接壓制，不進入連鎖
→ 低優先級能力的效果被完全蓋過

範例：
  時空法則能力 vs 火屬性能力
  → 時空法則直接壓制，火屬性能力無效
  → 不需要「反制」，效果自然不生效
```

### 11.2 第二層：同優先級的連鎖機制

當兩個能力優先級相同時，進入**連鎖堆疊**（參考遊戲王的連鎖機制）：

#### 基本規則：後發動者先結算（LIFO）

```
完整流程示例：

  [堆疊建立]
  A 宣告：「封印 B 的下一個能力」（層 1，進入堆疊）
  B 宣告：「無效 A 的封印」（層 2，疊在頂部）
  A 宣告：「無效 B 的無效」（層 3，疊在頂部）
  B 選擇不再連鎖 → 堆疊關閉，開始從頂部結算

  [結算，LIFO]
  層 3：A 的「無效 B 的無效」→ 結算成功，B 的層 2 被標記為無效
  層 2：B 的「無效 A 的封印」→ 已被無效，跳過
  層 1：A 的「封印 B 的下一個能力」→ 結算成功，封印生效

  結果：A 的封印生效，B 的反制失敗
```

#### MP 代價的支付時機

```
連鎖能力的 MP 在「宣告時」支付，而非「結算時」：

  → 宣告成功後，MP 立刻扣除
  → 若該層在結算時被無效化，MP 不退還
     （代價是宣告這個動作本身，不是效果是否成功）
  → 若因 MP 不足無法宣告，不得加入連鎖

設計意圖：MP 作為連鎖的門檻，防止隨意宣告連鎖再被無效後「白嫖」。
```

#### 連鎖的時機窗口

```
能力進入「啟動宣告」狀態後，至窗口時間到期前：
  → 雙方均可宣告連鎖（加入一層至堆疊頂部）
  → 每次有新層加入堆疊，雙方窗口重新計時
  → 雙方均選擇不連鎖，或窗口時間到期 → 堆疊關閉，開始結算

以下情況不可連鎖：
  → 對方沒有可用的連鎖能力
  → 對方 MP 不足以支付連鎖代價
  → 能力已進入「結算中」狀態（窗口已關閉）
  → 即時型能力（不可被連鎖）正在宣告中
```

#### 堆疊中的邊界情況

```
被無效的層仍然存在於堆疊中，只是結算時跳過：
  → 不會從堆疊中移除，不影響深度計數
  → 一個能力被無效後，無法再度被「無效的無效」復活

堆疊中同一個能力出現兩次（A 連段到同一個能力）：
  → 合法，兩層獨立結算
  → 各自消耗 MP，各自可被無效化，互不影響

若堆疊底部（層 1）在結算時被無效化：
  → 整條連鎖的「起因」消失，但上層已結算的效果不回滾
  → 已結算的層維持其效果（後發動先結算的原則）
```

#### 連鎖深度限制

「連鎖深度」指堆疊中從底部（第一個宣告的能力）到頂部（最後加入的能力）的總層數。

```
示例（深度計算）：
  層 1：A 發動「封印火元素」→ 進入宣告
  層 2：B 連鎖「無效 A 的封印」
  層 3：A 再連鎖「無效 B 的無效」
  層 4：B 連鎖「無效 A 的再無效」
  ...以此類推，直到某方無法或選擇不繼續連鎖
```

連鎖深度上限是**純系統機制**，與玩家持有的能力無關，玩家無法透過任何技能修改上限。
其存在目的是防止雙方無限互相連鎖導致遊戲卡死。

```
深度上限（⚠️ 數值待調整）：
  → 超過上限後，任何新的連鎖宣告均被系統拒絕
  → 系統通知雙方「連鎖已達上限，開始結算」
  → 從堆疊頂部向底部依序結算
```

### 11.3 第三層：持續規則的封鎖效果

**已生效的持續規則** 與 **連鎖中的能力** 性質完全不同：

#### 區分標準

```
連鎖中的能力：
  → 還在「宣告」到「結算」的中間狀態
  → 可以被連鎖干涉、無效化

已生效的持續規則：
  → 已完成結算，規則正在運行中
  → 不再是「能力」，而是「世界當前的規則」
  → 不可被連鎖，只能被更高優先級的能力覆蓋
```

#### 封鎖效果的運作

持續規則只能封鎖「優先級等於或低於封鎖者」的能力。
若發動者能力的優先級（法則等級或能量符文類型）高於封鎖者，則無視封鎖，直接進入宣告。

```
範例一（封鎖生效）：
  B 成功發動「封印半徑 50 米內的火元素能力」（持續規則，屬性刻印層）

  A 試圖發動火屬性技能（同為屬性刻印層，優先級 ≤ B）：
  → 系統判定：「當前規則禁止此類能力發動」
  → A 的技能連「宣告」資格都沒有
  → 不進入連鎖，直接失敗

範例二（優先級壓制封鎖）：
  B 成功發動「封印半徑 50 米內的火元素能力」（持續規則，屬性刻印層）

  A 試圖發動含火元素的基軸法則技能（法則層次高於屬性層）：
  → 系統判定：「發動者優先級高於封鎖者，封鎖無效」
  → A 的技能正常進入宣告，可被連鎖或直接結算
```

#### 持續規則的覆蓋條件

```
以下情況可以「打破」持續規則：

1. 更高優先級的能力（法則層次更高或能量類型優先）
   → 直接繞過封鎖，不受持續規則限制

2. 同優先級但針對「移除持續效果」的專屬能力
   → 進入連鎖，連鎖結算後才生效

3. 持續規則本身設有的解除條件
   → 條件達成時自然解除
```

#### 能力標籤系統（GameplayTags）

封鎖與條件判斷統一使用層級式標籤（GameplayTags），取代文字描述式規則：

```
標籤結構（層級繼承）：
  Ability.Fire           ← 所有含火屬性的能力
  Ability.Sustained      ← 所有持續生效型能力
  Status.Debuff.Frozen   ← 冰凍狀態（自動繼承 Status.Debuff 和 Status）
  Status.Buff.Haste      ← 加速狀態
  Element.Fire           ← 角色持有火屬性資格

查詢「Status.Debuff」會同時匹配所有子標籤（Frozen、Stun、Silence 等）

每個法陣設定（四種標籤欄位）：
  Ability Tags（本法陣的標籤）：「此法陣屬於 Ability.Fire」
  Required Tags（發動前置條件）：「發動者必須持有 Element.Fire」
  Block Tags（封鎖條件）：「若施法者持有 Status.Debuff.Silence，無法發動」
  Cancel On Tags（啟動時取消）：「啟動時取消施法者身上所有 Ability.Sustained 能力」

持續生效型封鎖的實作方式：
  施加「Block.Ability.Fire」標籤 →
  所有 Ability Tags 包含 Ability.Fire 且 Block Tags 設有 Block.Ability.Fire 的法陣自動無法發動
```

#### 持續效果疊加順序（Layer System）

當多個持續生效型效果同時修改同一個屬性時，按固定層序應用，防止結果因加入順序不同而不一致：

```
L1：類型替換效果（改變能力的基礎類型，如「火屬性 → 冰屬性」）
L2：能力封鎖/授予效果（Block Tags / 授予新能力）
L3：數值固定效果（設為固定值，如「傷害固定為 1」）
L4：加法效果（+X 傷害、+X 速度）
L5：乘法效果（傷害 ×Y）
L6：開關反轉效果（正傷害變負、治療變傷害）

同層內依「發動時間戳記」排序：後生成的效果後應用
依賴性例外：效果 A 的存在影響效果 B 的層別 → 先應用 A

靈感來自 MTG 的 Layer System。
```

### 11.4 特殊情況

#### 同一時刻多個能力同時觸發（非連鎖）

```
被動能力或條件觸發的能力可能在同一時刻觸發多個：

處理順序：
  1. 先比較優先級，高者先結算
  2. 同優先級：依能量類型決定順序（設計時定義）
  3. 同優先級且同能量：視為同一 tick 內的同時宣告，進入連鎖堆疊（見 §11.2）
     → 依 LIFO 結算，後加入堆疊者先結算
     → 即時型能力不進入連鎖堆疊，直接按觸發順序逐一結算
```

#### 持續規則 vs 持續規則的衝突

```
兩個相互矛盾的持續規則同時存在：

比較雙方的優先級：
  → 高優先級的規則有效，低優先級的暫時被壓制
  → 若優先級相同：後生效的規則優先（LIFO 原則延伸）

注意：被壓制的持續規則並未消失，
      一旦壓制它的規則解除，它重新生效
```

#### 能力「無效化」的範圍

```
「無效化」類型的能力需要在設計時明確指定範圍：

  無效化「下一個發動的能力」
  無效化「指定目標當前持有的所有能力」
  無效化「特定屬性的所有能力」（持續型）
  無效化「特定對象的下一次攻擊」

範圍越廣 → 消耗能力點越多 + 代價刻印越嚴苛
```

### 11.5 與遊戲王的對比

| 概念 | 遊戲王 | 本系統 |
|------|--------|--------|
| 連鎖 | 魔法/陷阱/效果怪獸的連鎖 | 同優先級能力的連鎖堆疊 |
| 結算順序 | 後發動先結算（LIFO） | 相同，後發動先結算 |
| 持續效果 | 持續魔法/場地魔法 | 持續規則，封鎖後續發動 |
| 無效化 | 「無效並破壞」 | 「無效化」效果刻印 |
| 優先級 | 相對固定（速度 1/2/3） | 動態，由法則層次和 MP 類型決定 |
| 不可連鎖 | 快速效果限制 | 持續規則不可連鎖 |

**主要差異**：本系統的優先級是玩家設計能力時就決定的，不是固定的「速度值」——這讓法則能量的選擇本身成為策略考量。

### 11.6 即時對戰的同時觸發定義（方案 B + C 混合）

#### 三種發動類型

本系統採用「能力重量決定窗口長度」的即時對戰設計，
三種類型代表**被反制風險**與**場面控制力**的取捨：

> ⚠️ 數值待調整（倍率 × 0.8 / × 1.0 / × 1.5 為草案）

| 類型 | 反應窗口 | 能力點倍率 | 能力上限 |
|------|---------|-----------|---------|
| **即時型** | 0ms，不可被連鎖 | × 0.8 | 禁用根源法則、持續效果 |
| **宣告型** | 100ms～1.5s（可設定） | × 1.0 | 無限制 |
| **持續生效型** | 無窗口（生效後封鎖） | × 1.5 + 持續 MP | 最強場地控制 |

#### 施放手感設計

三種類型在視覺和操作上有明確的「重量感差異」，讓玩家不靠 UI 就能辨認對手在用哪種類型：

```
即時型：
  不打斷角色移動（跑步中可施放）
  無蓄力動畫，點即發
  施放後無硬直

宣告型：
  有蓄力動畫（持續時間 = 反應窗口長度）
  蓄力中移速降低 30%
  可被打斷（打斷後進入短暫冷卻）

持續生效型：
  開啟時有明顯啟動動畫
  維持中角色手部持續特效，角色外觀有可見的「規則場域」視覺
  主動關閉有 0.5 秒收招動畫
```

#### 類型設定的兩種方式與優先序

發動類型可透過兩種機制設定，優先序如下：

```
優先序：邏輯刻印積木（動態）> 藍色圖騰改造刻印（靜態）> 預設（宣告型）

1. 邏輯刻印積木（動態，§7 發動類型積木）：
   在邏輯積木中使用「設為即時型 / 宣告型 / 持續生效型」積木，
   可根據條件在每次發動時動態決定類型。
   積木設定覆蓋藍色刻印的靜態設定。

2. 藍色圖騰改造刻印（靜態，設計時決定）：
   每次發動類型相同，適合單純固定用途的法陣。

   轉為即時型能力
   → 點數 × 0.8
   → 無使用限制
   → 不可被連鎖，但可以連鎖別人

   轉為宣告型能力（預設）
   → 點數 × 1.0，標準規則

   轉為持續生效型能力
   → 點數 × 1.5
   → 額外：每秒消耗固定 MP（維持費，依效果強度而定）
   → MP 耗盡時持續規則自動解除

   快速取消刻印（Roman Cancel）
   → 發動後立即額外消耗 50% 本法陣 MP，取消整個施放延遲
   → 允許立即接續下一個技能，不需等待施放延遲結束
   → 主要用途：製造「本應有後搖的重型法陣」直接接招的連段策略
   → 注意：下一個法陣仍有自身的施放延遲；§10 每幀執行次數上限仍然生效
   → 靈感來自 Guilty Gear 的 Roman Cancel 系統

   同名共享冷卻刻印（Hard Once Per Turn / HOPT）
   → 此法陣的冷卻改為「同名全局共享」：
     同一玩家裝備的所有相同名稱法陣，共用一個冷卻計時器
   → 任意一個觸發冷卻，其餘同名法陣同時進入冷卻
   → 防止「複製同名法陣多份輪流繞過冷卻」的設計漏洞
   → 能力點影響：不額外消耗（此刻印為自主約束，設計者決定是否採用）
   → 靈感來自遊戲王的硬性每回合限一次（Hard OPT）規則

   場景唯一次刻印
   → 本場戰鬥此法陣（含所有同名）共計只能宣告 N 次（N 由設計者設定，最低 1）
   → 達到次數上限後法陣灰化，無法施放，直到戰鬥結束
   → 與冷卻不同：不是時間限制，而是絕對次數上限
   → 能力點返還：次數 N 越小，返還越多（歸入限制換點機制，⚠️ 數值待調整）
   → 適合：「終極技能，每場戰鬥只能使用一次」的高峰設計
   → 暫定：每段發動前各別判斷；若中間段次數已用完，連段在此中止，前段已消耗 MP 不回滾。

3. 預設（無任何設定）：宣告型
```

#### 宣告型的反應窗口長度（接入限制刻印系統）

窗口長度是玩家在設計時決定的，可作為「限制換點數」的一個維度：

> ⚠️ 數值待調整（窗口長度區間與對應點數回報均為草案）

```
窗口長度            點數調整
─────────────────────────────
0.05 ～ 0.1 秒    + 3 點（幾乎無法反制，強力優勢）
0.1  ～ 0.3 秒    + 1 點
0.3  ～ 0.7 秒    ± 0（標準）
0.7  ～ 1.5 秒    - 1 點（主動給對手更多機會）
1.5 秒以上        - 3 點（明顯弱點，換取大量點數）
```

> 前搖越短 = 越難反制 = 消耗越多點數
> 前搖越長 = 越易反制 = 返還點數（等同自我施加限制）

#### 「同時觸發」的伺服器定義（方案 C 部分）

```
伺服器每 50ms 計算一次 tick

同一 tick 內收到的宣告型能力視為「同時觸發」：
  → 進入同一連鎖堆疊
  → 依優先級 + LIFO 結算

跨 tick 的能力：
  → 先 tick 的能力先完成宣告
  → 後 tick 的能力若在前者的反應窗口內，仍可加入連鎖
  → 超出反應窗口後宣告的能力，不得加入前者的連鎖
```

#### 客戶端預測（Network Prediction）

```
問題：玩家在本地按下技能，到伺服器確認結算有 50ms+ 延遲，
      等待確認才播放動畫會讓操作感覺遲鈍。

解法（參考 Unreal GAS 的 Prediction Key 模式）：
  1. 客戶端立即播放宣告型起手動畫（本地預測）
  2. 同時送出請求至伺服器驗證（附帶 Prediction Key）
  3. 伺服器驗證通過 → 繼續，結算效果
  4. 伺服器拒絕（MP 不足、封鎖標籤生效等）→ 客戶端回滾動畫，顯示「無法發動」提示

可預測（客戶端本地先行）：
  宣告動畫播放、UI 回饋、音效

不可預測（等待伺服器）：
  效果實際結算、HP 變化、新實體生成
```

#### 即時型的能力範圍限制

```
即時型無使用限制，所有刻印與效果類型均可裝備。
（注：極限法則刻印需額外 +2 點，此為能力點代價，非使用資格限制）
```

#### 持續規則的維持費

```
持續規則強制消耗 MP 維持：

基礎維持費：每秒 5 MP
效果強度調整：
  影響範圍每增加 10 米：+2 MP/秒
  封鎖的屬性或法則每增加一種：+3 MP/秒
  影響目標超過單體：+5 MP/秒

MP 耗盡時：持續規則立即解除，不觸發任何結算
```

#### 三種類型的戰略定位

```
即時型 → 突襲、搶先、打斷節奏
         便宜但效果單一，對手難以預判

宣告型 → 主力技能，可強可弱，有策略深度
         前搖設計本身就是資訊博弈

持續生效型 → 場地控制、封鎖策略
             成本高但壓制力強，構建圍繞它的戰術
```

### 12.1 執行流模型（主要來自 Noita）

#### 牌堆 / 手牌 / 棄牌模型（Deck / Hand / Discard）

Noita 的法杖執行本質上是一個卡牌遊戲：

```
牌堆（Deck）   → 待執行的圖騰序列
手牌（Hand）   → 當前正在執行的圖騰
棄牌（Discard）→ 本輪已執行完畢的圖騰

一輪結束後：Hand → Discard
牌堆清空後：觸發重置（Discard → Deck）並進入冷卻
```

**融入本系統的方式：**
- 圖騰法陣的圖騰可設定為「循環模式」，圖騰依序執行，執行完後進入棄牌區等待重置
- 「打亂模式」：每次重置後隨機排列執行順序（適合設計不確定性技能）

#### Shot State（執行上下文）

```
每次技能施放都建立一個暫時的「執行狀態物件」
所有圖騰和刻印都在修改這個物件，而非直接影響世界

執行狀態包含：
  - 當前傷害值
  - 擴散角度
  - 施放延遲累計
  - 投射物速度
  - 暴擊率
  - 穿透層數
  等等...

觸發容器執行時，建立一個「獨立的子執行狀態」
子狀態在執行上下文上獨立（各自維護傷害值、延遲累計等執行數據），
但以「施放時快照的施法者屬性資格」初始化（見 §4 屬性繼承）
```

**重要規則：觸發容器內的圖騰不貢獻父法陣的施放延遲。**

#### 施放延遲累計規則

```
圖騰法陣的施放延遲 = 所有圖騰延遲之和
但觸發容器（投射物/接觸）內的圖騰，延遲計入子狀態，不影響父法陣
負延遲可以抵消正延遲，但不可低於最小值（1幀）
```

#### 法陣重置（類比 Wand Refresh）

```
新增「法陣重置刻印」：
- 將所有已執行圖騰立刻移回牌堆
- 不觸發冷卻時間（這是與普通重置的關鍵差異）
- 不消耗圖騰的使用次數
- 可用來建立「儲存區」：
  法陣重置刻印之後的圖騰在正常流程中永遠不會觸發
  但可被特定機制（如「複製指定類型」刻印）存取
```

#### 常駐施放（Always Cast）

```
圖騰法陣有「常駐欄位」，與主序列完全分離：
- 每次法陣施放時，常駐欄位的圖騰免費附加執行
- 常駐圖騰的有限使用次數不會消耗（等同無限）
- 最多可設定 N 個常駐欄位（需消耗能力點解鎖）
```


#### 繞回執行（Wraparound）

```
當主序列圖騰不足以填滿「多重施放刻印」的需求時
自動從法陣頭部繞回繼續讀取
不觸發冷卻，繼續施放
直到下次牌堆清空才觸發正常重置
```

#### 修飾流動（Modifier Flow）

```
修飾刻印放在圖騰之前時：
  - 局部模式：只修飾緊接的下一個圖騰
  - 全域模式：修飾整個圖騰法陣內的所有同類圖騰

修飾刻印放在多重施放刻印之前時：
  → 該修飾以單次代價同時作用於所有多重施放的圖騰
  → 這是強力的「折扣放大」機制
```

#### 方向陣型（Formation）

```
多重施放刻印的進階版：
不只同時施放，還精確控制各投射物的角度分布

範例：
  六角陣型（6個，±60° 均分）
  扇形陣型（5個，指定角度範圍均分）
  環繞陣型（N個，360° 均分）
  前後陣型（2個，180° 相對）
```

### 12.2 資料類型與邏輯系統（主要來自 Psi）

#### 型別化連線（Typed Connections）

```
圖騰法陣中，節點間的連線是有型別的：

數值（Number）    → 浮點數
向量（Vector）    → 3D 座標或方向
實體（Entity）    → 單一生物或物件
實體列表（List）  → 多個實體
布林（Boolean）   → 真/假
任意（Any）       → 接受任何型別（用於轉接）

連線型別不符 → 編輯器顯示錯誤，無法儲存技能
```

#### 向量數學作為基本積木（Psi 最獨特的貢獻）

```
在邏輯刻印中加入以下向量運算積木：

建構/解構：
  向量建構（X, Y, Z → Vector）
  提取 X / Y / Z 分量

向量運算：
  加法、減法、乘以純量、除以純量、取反
  長度（magnitude）
  正規化（normalize，變成單位向量）
  叉積（cross product，得到垂直向量）
  點積（dot product，求投影）
  投影（project onto another vector）

空間查詢：
  射線投射（Raycast）：從位置沿方向投射，返回碰撞點
  軸向射線投射：限定於世界軸向的射線

實體轉向量：
  實體位置（→ Vector）
  實體朝向（→ Vector，面朝的方向）

這些積木讓玩家可以精確設計：
  追蹤型投射物（計算方向向量）
  幾何型能力（旋轉、環繞、反射軌跡）
  距離判定（比較向量長度）
```

#### 硬體分層（CAD 概念的類比）

Psi 的法術受四個獨立硬體組件限制，融入本系統為：

```
核心（Core）→ 法陣複雜度上限（計算步數預算）
插槽（Socket）→ 法陣規模上限（節點數量 / 同時啟用的法陣數）
電池（Battery）→ 能量池上限
構造（Assembly）→ 潛能上限（決定能使用哪些高階刻印）
               + 效率（決定實際能量消耗的比例）

潛能（Potency）與 效率（Efficiency）分開：
  潛能 → 能不能用某個刻印（門檻）
  效率 → 用了消耗多少能量（倍率）
  兩個是獨立的進升方向
```

#### 快取機制（Caching）

```
邏輯刻印中，如果同一個查詢積木的結果
被多個刻印同時使用，只計算一次，結果共享
（例如：「距離最近的敵人」查詢只執行一次，
 不管有多少刻印使用這個值）
避免重複計算，節省效能預算
```

#### 迴圈施放計數（Loopcast Index）

```
新增「施放計數」查詢積木：
返回此法陣從啟動以來執行的次數
可用於讓技能隨使用次數改變行為：
  第1次：做A
  第5次之後：做B
  第10次：觸發特殊效果
```

#### 成功計數（Success Counter）

```
新增「本次已成功執行的圖騰數」查詢積木：
可用於部分執行後的條件分支：
  如果前三個圖騰都成功了，才執行第四個
  否則觸發替代效果
```

#### 條件執行（Evaluate）與中止（Die）

```
條件執行積木（橙 → 布林輸入）：
  只有條件為真時，才執行包裹的圖騰
  類比 Psi 的 Trick: Evaluate

立刻中止積木：
  終止整個法陣的執行，後續圖騰全部跳過
  類比 Psi 的 Trick: Die
  用途：滿足某條件後不繼續執行（提前 return）
```

#### 延遲執行（Sleep）

```
等待 N 幀 / N 秒後繼續執行後續圖騰
法陣保持啟動狀態，期間持續消耗資源
類比 Psi 的 Trick: Sleep
```

#### 乾淨失敗（Fizzle）

```
技能施放失敗時不是靜默跳過，而是有明確分類的回饋：
  無法施放（前置條件未達，MP 未消耗）→ 顯示「無法施放：[原因]」
  Fizzle（執行途中目標失效，MP 已消耗）→ 顯示「目標已失效」
  兩種情況均不進入冷卻，失敗原因記錄於法陣錯誤日誌（設計介面可查，debug 用）。
  被斷招打斷屬另一套機制，不歸 Fizzle 管轄。
比靜默失敗更容易讓玩家理解技能邏輯。
```

#### 焦點目標（Focal Point）

```
角色視線終點作為一個基本的空間積木：
  焦點位置 → Vector（角色正在看的位置）
  焦點實體 → Entity（如果焦點正對著某個生物）

這讓玩家可以設計「指哪打哪」類型的技能
不需要複雜的目標選取條件
```

#### 時間回朔（Eidos Reversal）

```
對應第四層抽象能力（時間軸層）：

錨點刻印：儲存目標的當前狀態（位置、HP、buff）
回朔刻印：將目標狀態回滾到最近的錨點記錄

這是法則刻印「時間：回溯」的技術實作路徑
成本：極高能量消耗 + 嚴苛限制刻印
```

#### 預先計費投射物（Charge Bullet 概念）

```
投射物容器可設定「預先計費模式」：
  施放時付出能量，投射物延遲存在（可設定條件觸發）
  觸發時不再額外消耗能量
  允許同時釋放多個預置投射物
  策略性：提前佈局，條件成立時同時引爆
```

### 12.3 實體與狀態管理（主要來自 Scratch）

#### 克隆系統（Clone System）

Scratch 的克隆對應本系統的「召喚實體」：

```
召喚圖騰創建的實體（克隆）：
  繼承父實體的當前狀態（位置、外觀）
  有自己獨立的執行變數（不影響父實體）
  有自己獨立的圖騰法陣（裝載的能力）

克隆數量上限：由能力點決定上限
克隆可以再創建子克隆（多層召喚）
父實體死亡：子克隆可設定是否跟著消失
```

#### 全域變數 vs 實例變數（Variable Scope）

```
法陣變數有兩種作用域：

全域變數：
  所有法陣和召喚實體共享同一個值
  一個實體改變，所有實體看到的都改變
  適合：高利貸的「債務值」（全局追蹤）

實例變數：
  每個召喚實體有自己獨立的副本
  一個實體改變不影響其他實體
  適合：百足蟲「層數」（每個蟲各自計算）
```

#### 廣播訊息系統（Broadcast / Receive）

```
法陣間的通訊機制：

廣播（非同步）：
  發送訊號，立刻繼續執行，不等待回應
  所有訂閱該訊號的法陣都會收到並執行

廣播並等待（同步）：
  發送訊號，暫停執行，直到所有接收者完成才繼續
  注意：若接收者進入無限迴圈，發送者會永遠等待

用途範例：
  「擊殺事件」廣播 → 所有監聽「敵人死亡」的法陣觸發
  「玩家受傷」廣播 → 觸發被動防禦法陣
```

#### 狀態機（外觀/Costume 概念）

```
技能可以有多個「狀態」（類比 Scratch 的 Costume）：
  狀態 1：普通狀態
  狀態 2：強化狀態（觸發條件後切換）
  狀態 3：最終狀態（更嚴格條件）

切換狀態時：
  外觀改變（視覺特效不同）
  主動技能組可切換（類比你設計的「方案切換」）
  部分刻印效果可能依狀態不同而不同
```

#### 全域計時器（Timer）

```
系統提供一個全域持續運行的計時器（秒為單位）：
可用於技能中設定：
  「從技能開始後第 N 秒觸發 X」
  「在整場戰鬥的第 60 秒前有效」
  「每 30 秒重置一次」

比「持續 N 秒」更精確，適合需要絕對時間點的技能
```

#### 列表作為資料結構（List）

```
邏輯刻印中提供列表積木：
  加入元素到列表
  從列表刪除元素
  讀取第 N 個元素
  列表長度
  列表是否包含某元素

用途：
  追蹤受標記目標的列表
  記錄歷史傷害值（用於複利計算）
  管理多個召喚實體的狀態
```

#### 跨實體資料讀取

```
無需廣播訊息，直接讀取其他實體的變數值：
  「取得目標的 [變數名稱]」→ 返回數值
  非同步、不阻塞、永遠返回當前值

適合：
  讀取敵人當前 HP 判斷是否觸發效果
  讀取盟友的能量判斷是否需要援助
```

### 12.5 邊界情況與設計注意事項

#### 來自 Noita 的陷阱

```
1. 「施放數量」設定值不會觸發繞回
   只有明確的「多重施放刻印」才會
   → 設計時需明確區分兩種多重觸發機制

2. 觸發容器內的延遲不算進父法陣
   嵌套觸發鏈可以完全繞過施放延遲
   → 需要限制嵌套深度或設計相應代價

3. 法陣重置若出現兩次
   第二次觸發立即強制冷卻
   → 可作為設計「有限重置次數」的機制

4. 修飾刻印若法陣內沒有可修飾的圖騰
   修飾刻印會無限繞回試圖找到圖騰
   → 需要設計「修飾找不到目標時的行為」
```

#### 來自 Psi 的陷阱

```
1. 「乾淨失敗」是全局的
   一個條件不符會讓整個法陣停止，後續圖騰全部跳過
   → 玩家需要理解：失敗不是部分失敗，是全部失敗

2. 未連線的積木不執行，不報錯
   → 可能讓玩家困惑「為什麼這個積木沒有效果」
   → 介面上應標示「未連線的積木」

3. 型別不符在設計時報錯，而非執行時
   → 玩家可能花很多時間在調整連線型別
   → 需要清晰的型別視覺化

4. 範圍限制觸發乾淨失敗
   → 距離太遠的目標讓整個技能失效
   → 建議：範圍失敗只跳過該圖騰，而非整個法陣
```

#### 來自 Scratch 的陷阱

```
1. 「廣播並等待」死鎖：
   A 廣播給 B 並等待，B 也廣播給 A 並等待
   → 雙方互等，永遠不結束
   → 需要「廣播超時」機制：等待超過 N 秒自動繼續

2. 克隆創建時，全域變數是「共享的」而非複製
   若玩家預期克隆有獨立的全域計數，會出錯
   → 介面上需清楚區分全域/實例變數的圖示

3. 「不刷新畫面的快速執行」內若有無限迴圈，遊戲會凍結
   → 本系統以「施放延遲」作為最終防線：圖騰有最短施放間隔，確保不在單幀內無限執行

4. 克隆數量上限超過時，靜默失敗
   → 本系統應顯示「召喚實體數量已達上限」的提示
```

## 13. 介面設計方向

### 能力編輯器

```
[圖騰法陣命名欄]  ← 玩家可自由輸入任意名稱，強制顯示，不可省略
┌─────────────────────────────┐
│  [插槽 1]  [插槽 2]  [插槽 3] │  ← 插槽式排列（Phase 1 採用）
│  圖騰拖入插槽，刻印附掛在圖騰上  │
│  執行順序 = 插槽由左至右       │
└─────────────────────────────┘

左側面板：圖騰庫、刻印庫（按顏色分類）
右側面板：即時能力點消耗統計
下方面板：邏輯刻印積木編輯區（Scratch 風格）
```

> **Phase 1 決定**：採插槽順序（非節點連線圖）。圖騰依插槽位置順序執行，刻印直接附掛在所屬圖騰上，無需拖拉連線。節點連線（Psi 風格）留待未來視複雜度需求決定是否升級。

### 即時設計反饋

- **命名欄**：法陣名稱為強制欄位，玩家必須命名才可儲存；能力的個人化感從名字開始
- **即時消耗預覽**：每次拖入或移除刻印，右側面板即時更新能力點數字，不需施放才知道消耗
- **消耗上限警示**：接近或超過法陣消耗上限時，數字轉紅色並顯示「超出上限，請加入限制刻印」
- **限制刻印折扣對比**：加入限制刻印時同步顯示「加入前 X 點 → 加入後 Y 點（−Z 點 / 效果 −W%）」
- **鎖定圖騰提示**：玩家等級不足的圖騰顯示灰色鎖頭，懸停顯示「需玩家 LV XX 解鎖」
- **法陣風格傾向**：右側面板顯示當前法陣的刻印構成比例（如：傷害 70% / 控制 20% / 輔助 10%），幫助玩家意識到自己正在設計哪種風格的能力
- **圖騰 × 容器組合預覽**：將圖騰拖入容器插槽時，介面顯示「此組合的基礎行為說明」（如：觸發圖騰 + 投射物容器 = 「投射物命中時觸發廣域效果」），降低試錯成本
- **複雜度預算條**：設計介面顯示一條進度條，反映法陣所有積木和連線的計算開銷總和；接近上限時變色警告，讓玩家在設計時就知道能力的執行負擔（靈感來自 Dreams 的 Thermometer）
- **圖騰 × 屬性刻印兩維度分步選擇**：設計介面明確將「選圖騰（做什麼）」和「選屬性刻印（用什麼屬性做）」拆成兩個視覺上獨立的步驟；兩個維度各自有獨立的選擇區塊，強調它們是正交的決策，不混在同一個面板。（靈感來自 Elden Ring 的 Ash of War × Affinity 矩陣）

### 連鎖系統 UI（對戰中）

連鎖本為高難度操作，UI 只提供資訊，不降低操作門檻：

```
畫面顯示：
  對手宣告的能力名稱（或描述）
  剩餘宣告窗口的時間條（窗口越短，時間條消失越快，壓迫感越強）

玩家操作：
  連鎖反制透過鍵盤快捷鍵直接觸發，不彈出選擇介面
  能否及時連鎖取決於玩家對自身技能的熟悉程度與反應速度
  這是設計意圖，連鎖操作本身是技術層面的挑戰
```

### 能力隱密度系統

- 能力預設可被查看
- 消耗能力點提升隱密度
- 對方洞察度 > 隱密度：「部分識破」
- 對方洞察度 > 隱密度兩倍：「完全識破」
- 可自由開關隱密功能

### 召喚實體的能力欄位（新增）

- 能力可動態裝載至實體（Stealth Dolphin 機制）
- 裝載的能力有使用次數上限，使用後回歸原主

## 14. 技術選型

### 引擎

**首選：Godot 4**

```
原因：
- 同時支援 2D 和 3D，不需換引擎即可未來升維
- 完全免費開源
- 2D 物理和渲染對此類型遊戲夠用
- C# 支援（與 Unity 的知識可共通）
- GDScript 語法接近 Python，學習成本低
```

### 語言

```
C#（已定案）
能力系統的邏輯層不依賴維度，設計為引擎無關的純邏輯
```

> 選 C# 的理由：VM（ExecutionContext、BlockNode、型別系統）需要強型別與 interface，C# 比 GDScript 更適合；且邏輯層日後若抽成獨立函式庫也更容易。需額外安裝 .NET SDK 8.0。

### 世界物理

```
2D 階段：
  格狀模擬（Falling Sand，Noita 概念）
  中粒度 Tile（比 Minecraft 方塊小，比 Noita 像素大）——作為邏輯/物理單元
  次像素粒子（Sub-pixel particles）——作為視覺表現層

未來 3D 階段：
  替換為 Voxel 物理，邏輯層不變，表現層替換
```

### 粒度架構（模擬層 vs 渲染層分離）

核心設計：**模擬精度 ≠ 顯示精度**

```
模擬層（中粒度 Tile）              渲染層（次像素 / 粒子）
────────────────────────────────────────────────────────
Tile = 材質容器                →  每個 Tile 內以次像素粒子渲染
物理規則在 Tile 之間計算        →  視覺細節在 Tile 內展示
  液體：tile-to-tile 流動       →  流動視覺：sub-pixel 波動動畫
  火焰：tile 狀態傳播           →  火焰視覺：GPUParticles2D
  爆炸：摧毀半徑內 tiles        →  碎片：短命次像素飛散粒子
  技能效果：能力系統下指令       →  純 VFX，初期不參與物理
```

**次像素粒子說明**
- 粒子座標為浮點數（Godot 本身即浮點座標系），可精細到任意小數
- 多個次像素粒子可佔據同一顯示像素，其密度、溫度等屬性合併為像素顏色
- 顯示層下限仍是螢幕物理像素，但模擬精度可遠超像素等級
- 初期所有粒子視覺效果為純裝飾（不影響 Tile 物理狀態）
- 待 Tile 物理穩定後，再接入「粒子沉積→改變 Tile 材質」機制

| 粒子類型 | 模擬方式 | 渲染方式 |
|---------|---------|---------|
| 地形結構 | 中粒度 Tile 格 | 直接貼圖 |
| 液體流動 | Tile 細胞自動機 | Sub-pixel 波動（Shader） |
| 火焰蔓延 | Tile 狀態傳播 | GPUParticles2D |
| 爆炸碎片 | 一次性飛散計算 | 短命次像素粒子 |
| 技能特效 | 能力系統事件驅動 | 純 VFX，不參與物理 |

#### VM 最小可行版（Phase 1 第 4 步細項）

```
核心架構：
  ExecutionContext
    每次技能觸發建立一個實例，持有：
      當前執行的積木序列（BlockNode 佇列）
      變數 map（global / instance 分開儲存）
      執行狀態（執行中 / 完成 / Fizzle）
      消耗的 MP 累計（用於 Fizzle 退還計算）

  BlockNode
    每個積木的資料結構，包含：
      積木型別（enum）
      參數（填入欄位的值或變數引用）
      子節點（IF 的 then/else 分支、RepeatN 的循環體）
      輸入/輸出端口（型別標記，用於編輯器連線驗證）

  ExecutionLoop
    逐步執行 BlockNode，處理控制流跳轉（IF / Repeat / Wait）
    Wait 暫停時 context 保持存活，下一幀繼續執行

型別系統（Phase 1 僅實作基礎）：
  Number（浮點）
  Boolean
  Any（接受任何型別，用於未型別化的連線）

基礎控制流積木：
  IF（condition → then / else 分支）
  RepeatN（有限次數重複）
  Wait（等待 N 秒，context 保持存活）

效果與呼叫積木：
  EffectLabel    「此為效果『』」
  OnEffectStart  「當效果『』開始」
  OnEffectEnd    「當效果『』結束」
  InvokeSpell    「自身發動法陣『』」（呼叫外部法陣，按法陣名稱）
                 ── 連段執行規則 ──
                 → 發動前先執行 Pre-check（同施放失敗類型 1）：
                   MP 不足 / Required Tags 未滿足 / 目標不在範圍 → 連段中止，不消耗 MP
                 → 通過後扣除目標法陣的完整 MP，按其自身施放延遲執行
                 → 連段終止條件見 §3（MP 耗盡 / 條件不符 / 無下一法陣 / 斷招 / 連鎖無效化）
  InvokeTotem    「觸發法陣內圖騰『』」（呼叫當前法陣內具名圖騰）
  TotemDone      「圖騰『』已執行」（條件判斷）
  TotemHit       「圖騰『』命中成功」（條件判斷）
  TotemFizzle    「圖騰『』未命中 / Fizzle」（條件判斷）

變數積木（Phase 1 僅支援 Number）：
  SetVar（設定全域或實例變數）
  GetVar（讀取全域或實例變數）
  Compare（數值比較：>、<、=、≠）

施放失敗分兩類（性質不同）：

  ── 類型 1：無法施放（Pre-check 失敗，不進入宣告）──────────────
    前置條件驗證在 MP 扣除之前執行。
    觸發情況：
      MP 不足 / Required Tags 未滿足 / 目標不在範圍內 / 其他前置條件未達成
    規則：
      MP 不消耗
      技能不進入冷卻
      顯示「無法施放：[具體原因]」提示

  ── 類型 2：Fizzle（執行途中失敗，MP 已消耗）──────────────────
    前置條件已通過、MP 已扣除，但執行途中狀態改變。
    觸發情況：
      目標在執行途中消失 / 死亡 / 移出有效範圍
    規則：
      MP 不退還
      已飛出的投射物等保留，未觸發的部分跳過
      技能不進入冷卻
      顯示「目標已失效」提示

  ── 另：被斷招打斷（屬斷招刻印系統，非 Fizzle）─────────────────
    詳見「斷招機制」，不受上述規則管轄。

  共同規則：
    失敗原因記錄至「法陣錯誤日誌」（設計介面可查，debug 用）。

編輯器基礎驗證（設計時，非執行時）：
  型別不符：連線端口顏色不同，存檔前報錯並阻止儲存
  未連線積木：顯示警告標示（允許儲存，標記為「孤立積木」）
  內部圖騰引用不存在：存檔前報錯（圖騰名稱找不到）
  法陣循環呼叫警告：A 呼叫 B、B 呼叫 A 時提示（允許設計，僅警告）
```

## 16. 兩個子專案的分工

為了避免過早整合導致架構鎖死，分兩個專案平行開發：

### 介面合約（先定義，再分頭開發）

```
能力系統需要向世界詢問：
  getEntityAt(position) → 實體資訊
```

```
   → 做完後才能測試法陣實際效果

4. 能力平衡 / 刻印數值調整
   → 在 Mock 環境下迭代，確認後再接真世界

5. 子專案 B 世界系統完成後，替換 Mock World
```

子專案 B 可在 1–2 步驟期間獨立進行，不互相阻塞。

---

### Mock World 最小規格（格子場景）

**場景結構**
- 固定 20×20 格子地圖
- 每格三種狀態：空（可進入）、實心（不可進入）、可摧毀（可進入，可被 destroyTile 清除）
- 初始實體：玩家 ×1、假敵人 ×3–5（固定初始位置）

**每個介面的 Mock 實作方式**

| 介面 | 實作方式 |
|------|---------|
| `getEntityAt` | 查實體清單，按位置精確比對 |
| `getEntitiesNear` | 查實體清單，按歐氏距離過濾 |
| `getMaterialAt` | 查格子狀態（空／實心／可摧毀） |
| `getEntityProperty` | 每個實體儲存 HP、position、faction 三欄位 |
| `destroyTile` | 將格子標為已摧毀，fire `onTileDestroyed` |
| `applyForce` | 實體位置沿向量移動一格（忽略物理，純位移） |
| `setEntityProperty` | 直接改欄位值；HP 降至 0 自動 fire `onEntityDied` |
| `spawnEffect` | 僅 console log，不渲染 |
| `createEntity` | 加一筆新實體到清單 |

此規格足以測試：範圍型技能、位移技能、摧毀地形、連段觸發鏈、所有事件回呼。

## 17-A. 深度解析：五款核心參考遊戲

> 以下為深度研究後提取的具體機制、數字、名詞，  
> 每條「▶ 本系統應用」代表可直接優化/採用的設計決策。

### 1. Mages of Mystralia — 符文節點圖系統

#### 四種法術焦點（施放形式）

| 焦點 | 行為 | 類比本系統 |
|------|------|----------|
| **Immedi** | 瞬間近戰爆發 | 瞬間型圖騰 |
| **Actus** | 投射物（預設靜止，需 Move 符文飛行） | 投射物容器 |
| **Creo** | 召喚/地形創造（冰橋、火牆） | 召喚型圖騰 |
| **Ego** | 自身效果（護盾、衝刺） | 自身持續型圖騰 |

#### 三類符文（詳細名稱與成本倍率）

**行為符文（改變法術核心行為）：**

| 符文 | 魔力倍率 | 效果 |
|------|---------|------|
| Bounce | +100% | 反彈，不因反彈加速 |
| Detonate | +200% | 爆炸（配 Size 半徑×2） |
| Homing | +160% | 追蹤（配 Time 轉向速度×2） |
| Teleport | — | 施放起點移至目標位置 |
| Rain | — | 元素從高空落下 |

**擴增符文（附加在其他符文上，本身不增加主費用）：**

| 符文 | 效果 |
|------|------|
| Size | 魔力+200%，範圍×2 |
| Time | 魔力+200%，持續×2 |
| Inverse | 反轉方向（某些情況下*降低*成本）|
| Random | 隨機化路徑，魔力倍率減半 |
| Left / Right | 改變 Actus 飛行曲線方向 |
| **Ether** | **不產生任何效果，純增加 5 個連接節點** |

**觸發符文（if-then 邏輯，連結兩個完整法術）：**

| 符文 | 魔力倍率 | 觸發時機 |
|------|---------|---------|
| Impact | +150% | 碰到敵人或表面時 |
| Periodic | +150% | 每秒觸發一次 |
| Expire | +120% | 法術消失時 |
| At Once | +200% | 施放當下立刻同步觸發 |
| Proximity | +150% | 附近有敵人時持續觸發 |

#### 節點順序影響行為
```
「Actus + Right + Move」 ≠ 「Actus + Move + Right」
前者 = 直線向右飛
後者 = 飛行中途向右曲線

→ 符文的放置順序即程式執行順序，不是純加法
```

#### 玩家批評（值得避免的設計）
```
問題一：加了符文不知道魔力會增加多少，要施放後才看到 → 需要即時成本預覽
問題二：測試流程過繁：開選單→合成→關選單→施放→觀察 → 需要內建沙盒試驗模式
問題三：At Once（+200%）搭配連射效果過強且獲取過早 → 強力觸發符文需要能力點門檻
```

#### ▶ 本系統應用

```
1. Ether 符文概念 → 「空白延伸圖騰」：不帶任何效果，但在法陣中
   提供額外 4 個連接插槽，讓複雜法陣可以展開更大的結構

2. 擴增符文「不增加主費用，附加在其他刻印上」→
   本系統的「修飾刻印」掛載在「效果刻印」上，費用由宿主刻印承擔

3. 觸發符文的五種時機 → 直接對應觸發型圖騰的觸發條件選項：
   On Hit（命中時）/ Timed（計時）/ On Expire（持續結束）/
   Immediate（施放當下）/ Proximity（感知距離內）

4. At Once（施放同時觸發子法術）= 本系統宣告型能力的「即時連鎖」選項，
   成本最高（+200%）→ 對應即時連鎖的高能力點消耗

5. 設計師介面必須實作「即時能力點消耗預覽」，
   每次拖入刻印立即更新顯示，不讓玩家靠猜測決策
```

### 2. Tyranny — Sigil 四層法術組裝

#### 四層 Sigil 對應表

| 層級 | 數量 | 功能 | 類比本系統 |
|------|------|------|----------|
| Core Sigil | 11種 | 魔法元素（火/冰/雷/力/衰/生/情/石/月/幻） | 能量刻印（MP類型） |
| Expression Sigil | 9種 | 投遞形式（決定法術形態） | 圖騰類型 |
| Accent Sigil | 可疊多個 | 數值修飾、附加狀態 | 屬性刻印、控制刻印 |
| Enhancement Sigil | 限 1 個/法術 | 特殊規則（觸發條件/目標優先級等） | 邏輯刻印（每法陣限一） |

#### 解鎖門檻：Lore 值

```
Expression Sigil 要求 Lore 值（施法知識等級）：
  Focused Intent（近戰單體）   需 Lore 15
  Distant Impact（遠程投射）   需 Lore 30
  Channeled Strength（錐形）   需 Lore 40
  ...
  Chaotic Descent（廣域隨機）  需 Lore 100

→ 不是「花多少點數」，而是「你的知識等級夠不夠」
```

#### 最有創意的兩個設計

**Sigil of Frostfire（冰火融合）：**
```
同時含有冰屬性和火屬性
施放時自動偵測目標抗性，選擇目標較弱的那一側傷害
代價：效果略弱於純種元素（但天然消除元素免疫問題）
```

**Sigil of Blood Magic（血魔法）：**
```
效果：法術傷害大幅提升
代價：每次施放對自身造成原始傷害（至少 15 點，約 33%）
→ 代價不是資源，而是傷害自身（高風險高報酬）
```

#### 堆疊限制

```
相同 Expression Sigil 效果不疊加（兩個 Focused Intent 不累加）
不同 Expression Sigil 的效果可共存（護盾 + 攻擊增益同時生效）
每個法術只能含一個 Enhancement Sigil
```

#### ▶ 本系統應用

```
1. Lore 門檻機制 → 圖騰等級門檻：
   特定圖騰不是「買了就能用」，而是角色「領悟等級」需達到閾值
   高階圖騰（如置換型、領域型）需要玩家 LV 50+

2. Frostfire 融合概念 → 雙能量融合刻印：
   同時含兩種 MP 類型的刻印，施放時自動選擇目標抗性較低的那種傷害
   設計為「適應型」能量，略弱於純種但消除弱點免疫問題
   需兩種 MP 同時滿足發動條件

3. Blood Magic → 自傷代價刻印（已有，但要明確支援「固定傷害」vs「比例傷害」）：
   限制刻印新增參數：「代價 = 對自身造成 N 點 / 當前血量 X% 傷害」

```

### 3. Morrowind Spellmaker — 參數精細化與公式設計

#### 效果參數模型

```
每個法術效果獨立設定四個參數：
  最低威力 (Min Magnitude)  → 施放時取隨機值
  最高威力 (Max Magnitude)  → 施放時取隨機值
  持續時間 (Duration)       → 最高 1440 秒（= 24 分鐘）
  範圍半徑 (Area)           → 最高 50 英尺

→ 不是固定值，而是一個「隨機範圍」：每次施放結果略有差異
```

#### 魔力消耗公式

```
效果費用 = (2 × 平均威力 × (持續+1) × 基礎費用/40) + (面積 × 基礎費用/40)
遠程施放 × 1.5
購買費用 = 魔力費用 × 7 金幣

施法成功率：
(技能×2 + 意志/5 + 幸運/10 − 法術費用 − 聲音影響) × (0.75 + 0.5 × 當前疲勞/最大疲勞)
```

#### 多學派法術的學派歸屬規則

```
同一法術含多個魔法學派的效果時，
法術歸屬於「玩家施放成功率最低的那個學派」

效果一：練習弱項的捷徑（故意放入弱學派效果刷 XP）
效果二：意外讓高消耗法術更難施放（未料到的懲罰）
```

#### 著名漏洞（設計反例）

```
費用溢出漏洞：
  若法術理論費用超過 65535（遊戲最大值），費用從頭循環
  → 理論費用 65542 的法術實際費用只需 6 魔力
  → 官方從未修補

自我強化無限循環：
  「強化鍊金術 2 秒」→ 製作超高等藥劑 → 提升所有屬性 → 再強化鍊金術...
  理論上沒有上限
```

#### ▶ 本系統應用

```
1. 效果參數採「範圍值（最低～最高）」而非固定值：
   基礎效果刻印允許設定傷害範圍（如 200%~300%），
   施放時在範圍內隨機，增加不確定性和趣味，
   可用「穩定性刻印」收窄範圍（代價：增加能力點消耗）

2. 施法成功率受狀態影響 → 施放穩定性機制：
   疲勞/失去平衡/中毒/恐懼等狀態影響宣告型/持續型的施放穩定性，
   不是「能否施放」，而是效果數值打折（稍微降低威力或持續時間）

3. 多 MP 類型混合能力的「主導 MP」規則：
   若一個能力同時使用多種 MP，
   主導 MP = 點數消耗最高的那種，
   能力的優先級層級由主導 MP 決定

4. 費用溢出防護 → 能力點上限硬封頂：
   能力點消耗不採用開放公式，而是設定單一法陣消耗上限（如 LV100 = 最多 500 點），
   超過上限自動灰階（無法完成），提示玩家需先取得限制刻印

5. 自訂命名功能保留：
   法陣（能力組）必須允許玩家自訂名稱，
   這是強烈個人化感的重要來源（Morrowind 老玩家最懷念的功能之一）
```

### 4. Magicmaker — 插槽並聯 × 施法力縮放

#### 三件道具的差異化設計

| 道具 | 施法力 | 魔力消耗 | 效果範圍 |
|------|--------|---------|---------|
| **法杖 (Wand)** | 10%（法術的 1/10） | 無 | 主動攻擊輔助 |
| **法術 (Spell)** | 100%（基準） | 消耗魔力 | 主要攻擊 |
| **法袍 (Robe)** | 0（無施法力） | 無 | 純被動效果 |

```
同一材料在不同道具上效果不同：
  Lightning Rod 放法術 → 電弧傳遞至鄰近敵人
  Lightning Rod 放法袍 → 行走時充電，偏轉敵方投射物

→ 相同組件，容器不同，效果完全不同（容器本身是設計維度）
```

#### 施法力縮放機制

```
每個材料有「施法力影響比例」：
  例：毒素材料 → 毒傷的 50% 受施法力縮放

Philosopher's Stone A 級（法術插槽）：
  施法力 100 → 350（+250%）
  毒傷提升 = 250% × 50% = +125%

Enchantment 類型（特殊附魔模式）：
  直接傷害 = 0
  施法力 = 正常的 3 倍
  → 使只依賴施法力縮放的材料效果暴增（逆向設計）
```

#### 插槽數量成長（NG+ 系統）

```
初始：法術 1 格、法杖 1 格、法袍 1 格（= 共 3 格）
每次 NG+ 某特定關卡完成後增加一格（三件各自獨立成長）
最終上限：7 格/件（= 共 21 格，需 NG+++ 以上）

→ 成長感來自「同樣的材料，但可以同時放更多種」，不是解鎖新材料
```

#### 開發者自評的遞迴問題

```
某些材料（如「讓螢幕充滿彈丸」類型）接管整個法術，
其他材料效果被淹沒，插槽失去決策意義。

設計初衷：「每個材料插槽都是決策點」
實際問題：有些材料太強，壓制其他所有插槽的存在感
```

#### ▶ 本系統應用

```
1. 「施法力縮放比例」→ 每個刻印有固定縮放係數：
   每個效果刻印有個別縮放係數，決定投入能力點後效果提升的幅度，具體公式待實作時確定。
   概念：「裂傷刻印」縮放係數低（多加點數邊際效益小），
         「核爆刻印」縮放係數高（高投入高回報，但基礎消耗極高才能達到）

2. Enchantment 概念 → 「增幅型法陣」：
   法陣可宣告為增幅型，自身不造成直接效果，
   但使同角色其他能力的效果係數 ×3
   代價：此法陣不可有傷害類刻印，且持續消耗 MP

3. 相同組件在不同容器效果不同：
   本系統已有三種執行容器（實體/投射物/接觸），
   刻印在不同容器中行為可設定為「容器自適應」：
   如「電弧刻印」在投射物容器 = 命中後電弧傳遞，
                在接觸容器 = 接觸的瞬間電弧爆發

4. NG+ 插槽成長 → 不採納：法陣刻印插槽數量固定，不隨能力點成長。

5. 遞迴材料問題的預防：
   每個法陣的「單一刻印強度上限」= 法陣總能力點的 60%
   若某一刻印超過這個比例，系統警告「此刻印主導法陣，其他刻印可能無法發揮」
   玩家可忽略警告（允許高度特化），但系統透明化揭示設計問題
```

### 5. Spoxel — 線性積木序列與非戰鬥法術

#### 符文積木三類

```
目標選取符文 (Targeting Runes)：
  Projectile Targeting — 投射物
  AoE Targeting       — 範圍爆炸

效果符文 (Effect Runes)：
  Fire Damage / Dig / Shield / ...（效果本身）

變異符文 (Mutator Runes)：
  Shape Mutators    — 改變形狀
  Change Target     — 變更目標類型
  Bounce            — 反彈
  Negative Mutators — 降低魔力消耗，但削弱效果（負向取捨）
```

#### 法術書與快捷鍵

```
法術綁入法術書，設定兩個滑鼠按鍵 + 最多 3 個快捷鍵
→ 玩家可在法術書中預先儲存多組配置，戰鬥中快速切換
```

#### 非戰鬥法術的重要性

```
玩家可設計：
  挖礦法術（Dig 效果 + AoE + Shape = 大面積採掘工具）
  建築法術（放置方塊 + 目標定位）
  移位法術（抓取敵人投擲至熔岩坑）

→ 同一套積木系統同時服務戰鬥和生產，玩家可以「用能力代替工具」
```

#### ▶ 本系統應用

```
1. 負向變異符文 → 代價刻印明確標示效能折扣比例：
   目前限制刻印已有「換取便宜」的概念，
   強化這一點：每個限制刻印在設計介面顯示
   「此刻印使能力點消耗降低 X 點，但效果削弱 Y%」
   玩家清楚知道在做什麼取捨

2. 法術書快速切換 → 能力組預設集（Loadout）系統：
   法陣不只是一個能力，而是一組快速切換配置
   玩家可儲存多個「技能組」（如戰鬥組、生產組、探索組），
   戰鬥外可隨時切換整套技能配置

3. 非戰鬥法術 = 本系統的職業技能延伸：
   能力系統支援非戰鬥效果刻印（採集、合成、建造、偵查），
   玩家可製作「採礦爆破法術」「高速建造法術」等工具型能力
   這些能力不需要 MP 即時型設計，可以改用「充能型」（使用次數限制）

4. 玩家反饋的教訓（Spoxel 失敗原因）：
   系統易被濫用導致平衡崩潰 → 非戰鬥法術和戰鬥法術的能力點池應分離
   玩家的採礦法術不佔用戰鬥能力點資源，
   防止「把所有能力點砸進無限挖掘」破壞世界生態
```

### 1. Path of Exile — 插槽連結 × 觸發寶石 × Keystone × Curse

#### 插槽與連結系統（Socket & Link）

```
插槽顏色與屬性對應：
  紅槽 → 力量技能
  綠槽 → 敏捷技能
  藍槽 → 智力技能
  白槽 → 任意顏色

最大插槽數：
  胸甲、雙手武器 → 最多 6 個插槽（終局追求：6-Link）
  頭盔、手套、靴子 → 最多 4 個插槽
  單手武器、盾牌 → 最多 3 個插槽

6-Link 核心組成：1 個主動技能 + 最多 5 個輔助寶石
```

#### 輔助寶石標籤相容系統

```
每個技能有標籤（如 Melee、Projectile、Spell、AoE...）
每個輔助寶石有「支援要求標籤」

例：
  Multistrike Support → 只支援含 Melee 標籤的技能
  Greater Multistrike → 費用倍率 160%，近戰重複 3 次
  Added Fire Damage  → 費用倍率 120%，25-39% 物理轉火焰

相同輔助寶石不疊加：兩顆相同輔助只取等級較高者
```

#### 費用乘法疊加

```
多個輔助寶石的費用倍率相乘（非相加）：
  3 個各 120% 的輔助：技能費用 × 1.2 × 1.2 × 1.2 = 原本 173%

→ 大量輔助導致費用暴增，是天然的複雜度熔斷器
```

#### 觸發寶石（Trigger Gems）— 最關鍵的新發現

| 觸發寶石 | 觸發條件 | 效果 | 限制 |
|---------|---------|------|------|
| **Cast When Damage Taken (CWDT)** | 受到足夠傷害 | 自動施放連結的法術 | 不支援圖騰/陷阱/地雷，CD 0.25 秒 |
| **Spellslinger** | 揮動法杖射擊時 | 同時觸發所有連結法術 | 保留魔力啟用（佔用最大魔力%） |
| **Mirage Archer** | 擊中敵人時 | 在頭頂召喚幻影複製同一技能 | — |

#### Keystone 節點（里程碑）— 改變規則的特殊節點

每個 Keystone 有一個強大正面效果 **+** 一個明顯負面效果，迫使玩家圍繞它重新設計整個方向：

| Keystone | 正面效果 | 負面效果 |
|----------|---------|---------|
| Resolute Technique | 100% 命中率 | 無法暴擊 |
| Chaos Inoculation | 混沌傷害完全免疫 | 最大生命值變為 1 |
| Pain Attunement | 低生命時 +30% 暴擊傷害 | 滿生命時 -30% 暴擊傷害 |
| Ancestral Bond | 可召喚無限圖騰 | 每個圖騰消耗 100 Spirit |
| Mind Over Matter | 40% 傷害由魔力承受 | 魔力回復 -50% |

#### 詛咒系統（Curse）

```
Hex（咒術）：
  範圍施放，對 Boss 效果大幅削弱（地圖 Boss -33%，Sirus -66%）
  在目標身上逐漸累積 Doom 值，Doom 越高效果越強
  可施加多個（上限由技能點提升）

Mark（標記）：
  單體施放，永久附著直到移除或換目標
  同一時間只能有一個 Mark 生效，即使增加詛咒上限也不例外

Impending Doom Support（倒數引爆）：
  連結 Hex 時，Hex 到期或被移除時爆發混沌傷害
  傷害量基於到期前累積的 Doom 值 → 「疊 Doom 再爆發」玩法
```

#### ▶ 本系統應用

```
1. 觸發型刻印（CWDT 概念）→ 被動反應刻印：
   一種特殊刻印：「受到 X 點傷害時，自動施放指定法陣」
   此刻印不佔用主動技能欄，掛在被動槽中，
   給予防禦型玩家「被打就反擊」的被動設計空間

2. Keystone 概念 → 關鍵圖騰（Keystone 圖騰）：
   特殊的高能力點圖騰，帶有一個強大正面效果和一個強制負面效果
   例：「無界圖騰」— 能力射程無限 / 但施放後固定站在原地 2 秒
   玩家必須圍繞這個負面效果重新設計整個法陣

3. Hex Doom 累積 → 層疊計時引爆刻印：
   「詛標刻印」：在目標身上持續累積「Doom 層」（每秒 +1 層），
   最高累積 5 層；若 Doom 到期前未被引爆，自動爆發
   「引爆刻印」：主動觸發目標所有 Doom 層，傷害按層數倍增

4. Hex（範圍詛咒）vs Mark（單體標記）的雙軌設計：
   本系統的刻印可標記為「廣域型」（對範圍內所有敵人生效，效果稍弱）
   或「單體型」（只對一個目標生效，效果強，但只能同時有一個目標）
   兩者共存，給群控型和精殺型玩家不同的設計路線

5. Spellslinger 概念 → 武技連動刻印：
   「連動刻印」：啟動後，每次普通攻擊或武技圖騰命中時
   自動同步施放指定法陣，無需獨立觸發，代價是保留一定比例 MP
```

### 2. Magicka — 元素優先級 × 物理直覺 × 命名法術

#### 8 基礎元素 + 3 複合元素

| 按鍵 | 元素 | 性質 |
|------|------|------|
| Q | Water（水） | 使目標「濕透」，濕透後受閃電傷害大幅提升 |
| W | Life（生命） | 光束型治療，主要用於自身 |
| E | Shield（護盾） | 防護屏障，不能與自身疊加 |
| R | Lightning（閃電） | 對濕透目標傷害倍增，鏈式傳導 |
| A | Earth（大地） | 投射物型，與幾乎所有元素相容 |
| S | Fire（火焰） | 噴射型，持續燃燒 DOT |
| D | Cold（冰冷） | 減速，疊加越多範圍越大 |
| F | Arcane（奧術） | 光束型，最通用，持續時間隨疊加量增加 |

**3 個複合元素（兩種基礎組合生成）：**
- Water + Cold → **Ice（冰）**：固態投射物
- Water + Fire → **Steam（蒸汽）**：蒸汽光束/噴射
- Water + Arcane → **Poison（毒）**：毒素效果

#### 優先級系統（決定法術形態）

```
當佇列中混合多種元素，由優先級最高的元素決定法術「形態」：

護盾 > 投射物（Earth、Ice）> 光束（Life、Arcane）>
蒸汽 > 閃電 > 噴射（Water、Fire、Cold）

例：Arcane + Water 的結果 = 奧術光束（並附帶濕透效果）
   而非 水噴射（附帶奧術加成）
→ 高優先級元素決定形態，低優先級元素決定附帶效果
```

#### 相消規則（禁止組合）

```
Fire ↔ Cold     互相抵消（兩者都消失）
Fire ↔ Water    互相抵消
Lightning ↔ Water
Lightning ↔ Earth
Life ↔ Arcane
Steam ↔ Cold / Ice ↔ Fire
Shield 佇列中只能有一個
```

#### 命名法術（Named Magicks）

```
22 個命名法術需收集法術書後解鎖，效果遠超普通元素組合：
  Revive         = W + R（Life + Lightning）  → 復活隊友
  Haste          = A + S + F                 → 移速翻倍 10 秒
  Conflagration  = 8 個元素特定序列           → 600 火焰範圍傷害
  Napalm         = Q + F + D + W + F + F     → 呼叫飛機燃燒彈空襲
  Thunder Storm  = 特定序列                  → 召喚閃電風暴

→ 「普通組合發現」之上，還有「命名法術發現」的另一層驚喜感
```

#### ▶ 本系統應用

```
1. 優先級決定形態（不只影響傷害）→ 圖騰類型競爭規則：
   同一法陣中若同時有「投射物圖騰」和「領域圖騰」，
   不是兩個都運作，而是「較高等級的圖騰決定能力的主要形態」，
   另一個圖騰降格為「附帶效果層」（效果減半，但不浪費）

2. 物理直覺設計（濕透→閃電倍增）→ 本系統屬性碰撞強化直覺性：
   現有的屬性碰撞表已有（如 水+雷→感電），
   但可進一步強調「前置狀態放大後置效果」的直覺邏輯：
   「已被水濕透的目標受閃電傷害時傷害 ×2」
   讓玩家自然理解「應該先濕後電」，而非靠背公式

3. 命名法術系統（Magicka 特有，不採納）：
   Magicka 的「特定組合 → 系統辨識 → 給予隱藏稱號」設計與本系統的設計哲學衝突。
   本系統不設「等待發現的隱藏刻印配方」；
   特殊稱號僅來自玩家的創作行動里程碑（首次命名法陣、圖騰數量達標等），
   不依賴系統自動辨識刻印組合。

4. 4 種釋放方式（右鍵/Shift+右鍵/中鍵/空格）→ 不採納：
   本系統已有三種執行容器（投射物/接觸/召喚實體）。
   「以施法者為中心的 AoE」由現有觸發圖騰的「周身」形態覆蓋，不需新增第四種容器。
```

### 1. Baba Is You — 規則即物件 × NOT 覆蓋 × 屬性疊加

#### 核心機制

```
語法結構：[名詞] IS [屬性/名詞]，文字方塊在地圖上物理存在
動詞（12 種）：IS、HAS、ISNOT、MAKE、BECOME、EAT、FEAR、FOLLOW、MIMIC、PLAY、WRITE、ISIN
運算子：AND（疊加）、NOT（否定）

衝突解決：
  NOT X 永遠優先於 X（NOT PUSH 使 PUSH 失效）
  雙重否定：NOT NOT X = X（互相抵消）
  矛盾規則：HEDGE IS NOT HEDGE → 所有 HEDGE 物件自我毀滅

屬性疊加（Stacking）：
  大多數屬性可共存，不互相排除
  AND 顯式疊加放大效果：
    SHIFT AND SHIFT → 每回合推進 2 格（非 1 格）
    BOOM AND BOOM AND BOOM → 爆炸半徑逐級擴大

元規則（Meta Rules）：
  LEVEL IS BABA → 整個關卡本身變成 Baba 物件
  TEXT IS X → 所有文字本身獲得屬性

性能優化：
  系統只在文字方塊移動或改變時才重新計算規則（懶惰評估）
```

#### ▶ 本系統應用

```
1. 效果共存設計原則：
   大多數刻印應設計為「可共存疊加」而非「互相排除」；
   只有在明確設計為衝突時（如 HOT 和 COLD 同類屬性）才互相否定。
   這是刻印疊加機制的基本哲學，確認現有設計方向正確。

2. NOT 打消刻印 → 新型紅色侵略刻印子類「打消刻印」：
   指定一種效果類型，消除目標身上已生效的該類型效果
   與「無效化刻印」的差異：無效化 = 阻止能力宣告；打消 = 移除已生效的效果

3. AND 疊加放大 → 確認刻印疊加規則：
   同一法陣內允許多個同類效果刻印（AND 疊加），效果值累加，不覆蓋
   但同一法陣的同類刻印有數量上限（防止純粹疊加破壞平衡，數值待定）

4. 規則引擎懶惰評估（性能優化）：
   被動反應觸發積木只在「條件狀態發生改變」時重新評估
   而非每幀遍歷所有被動條件——直接對應邊緣觸發積木的設計意圖
```

### 2. Opus Magnum — 三維度優化 × 指令集 × 廢棄口

#### 核心機制

```
評分三維度（Pareto frontier，三者互相取捨）：
  Cost（黃金，g）：所有機構的建造成本
  Cycles：完成一次產品輸出所需的時鐘週期
  Area：解決方案佔用的六邊形格子數
  （社群第四指標：Instructions，指令總數）

手臂指令集（每個手臂最多 24 條指令）：
  grab / drop / rotate_cw / rotate_ccw / pivot_cw / pivot_ccw
  extend / retract（僅活塞手臂）
  track_plus / track_minus（僅配合軌道）
  repeat / reset / noop

廢棄口（Disposal Glyph）：成本 0g，刪除不需要的原子
noop：空等待一拍，是合法的時序設計工具

平行執行：所有手臂在同一 Cycle 執行各自指令，碰撞以 0.1 間隔做 10 次分段偵測

社群技術：
  Overlap Cycles：多手臂作業週期在時間上重疊，壓縮總 Cycle 數
  Syrup Tech：分子卡住後透過重新鍵結回收，無需浪費
```

#### ▶ 本系統應用

```
1. 三維度設計評估 → 設計介面右側面板同時顯示法陣的三個向量：
     能力點（對應 Cost）
     施放延遲（對應 Cycles）
     刻印數量 / 複雜度（對應 Area / Instructions）
   讓玩家意識到「強化某一維度往往犧牲另一維度」，是一個清晰的設計反饋機制

2. Null 廢棄口 → 已在 §7 新增「捨棄積木（Null 輸出）」
   成本為 0，接受任意型別輸入並忽略
   用途：積木的輸出端口必須連線但玩家不需要該結果時使用

3. 指令數上限（24 條）→ 設計時複雜度上限的參考：
   每個法陣的邏輯刻印積木不設執行步數上限（已移除），
   但介面複雜度預算條可以反映「積木指令總數」作為視覺提示（非強制限制）

4. noop 是合法設計工具：
   Wait 積木不只是被動等待，也是主動的時序設計工具——
   刻意在連段中插入等待，製造節奏感或等待條件成熟
```

### 3. SpaceChem — Waldo 優先序 × Flip-Flop × 管線緩衝

#### 核心機制

```
紅 Waldo 永遠優先於藍 Waldo（每 Cycle 紅先執行）：
  確定性規則，消除「同時觸發」的歧義

Flip-Flop 指令：
  每次 Waldo 經過此指令時，在兩個方向之間交替
  → 奇數次走路線 A，偶數次走路線 B
  → 自然實現「每隔一次做不同事」的模式

Sync 指令：
  Waldo 到達 Sync 後暫停，直到另一顏色 Waldo 也到達 Sync 才同步繼續
  → 顯式同步點；替代方案：精心調整兩個 Waldo 的路線長度，讓它們自然同步

同 Cycle 傳遞：紅 Waldo 放下 → 藍 Waldo 同 Cycle 拾取（無需等待下一 Cycle）

管線緩衝：
  管道長度 = 分子延遲量
  下游反應器較慢時，加長兩者之間的管道提供緩衝，避免上游阻塞

Sensor 指令（特殊反應器）：
  偵測通過 Spectrometer 的原子種類，依類型分叉路線
```

#### ▶ 本系統應用

```
1. 固定優先序解決同時觸發問題 → 已在 §11.4 採用「先觸發者先結算」
   SpaceChem 的「紅優先於藍」是最簡單可靠的確定性解法
   本系統等效：被動/條件觸發能力在同一 tick 內，依觸發時序排序，無需動態屬性決定

2. Flip-Flop 交替觸發 → 已在 §7 新增「交替觸發積木」
   每次觸發在 A 和 B 之間輪流執行
   典型用途：奇數次攻擊觸發傷害效果，偶數次觸發強化效果

3. 管線長度作為設計工具 → Wait 積木的設計定位強化：
   連段中的 Wait 不只是安全機制，也是刻意的時序設計工具
   相當於 SpaceChem 用管線長度控制兩個反應器的同步節奏

4. 自然同步優於顯式同步：
   大多數情況下，透過 MP 消耗和 Wait 積木的精確設計可以讓法陣自然同步
   BroadcastAndWait（顯式同步）是最後手段，不是首選
```

### 4. TIS-100 — ANY 埠 × NIL 暫存器 × 鉗位算術

#### 核心機制

```
節點類型：
  T21（基本運算節點）：最多約 15 行指令，循環執行
  T30（堆疊記憶體節點）：容量 15 個值，LIFO

暫存器：
  ACC（-999 到 999，所有算術的目標）
  BAK（備份，只能透過 SAV/SWP 存取）
  NIL（讀取恆為 0，寫入忽略 = 垃圾桶）

特殊埠：
  ANY：讀取時接受最先準備好的埠（優先序：LEFT→RIGHT→UP→DOWN）
       寫入時送往最先準備好的埠
  LAST：指向 ANY 最近一次使用的具體埠

關鍵特性：所有埠操作都阻塞（blocking），直到通訊雙方都準備好才繼續

算術鉗位：ADD/SUB 結果鉗位在 ±999，不溢出不迴繞

死鎖陷阱：A 等待從 B 讀，B 等待從 A 讀 → 永久卡住
節點更新順序不保證，不能依賴特定順序
```

#### ▶ 本系統應用

```
1. NIL 暫存器 = Null 輸出積木（已在 §7 採用）：
   讀取恆為 0 且寫入無效的設計，正好是「捨棄積木」的精確規格

2. 算術鉗位規範：
   VM 的 Number 型別在設定的合法範圍內鉗位（不溢出不迴繞）
   各類積木的輸出範圍在設計時明確定義：
     HP% 積木 → 鉗位 0–100
     傷害值 → 鉗位 0–上限（防止負傷害）
     角度 → 鉗位 0–360 或 ±180

3. ANY 埠 = 廣播系統的「接受任意訊號」接收端：
   當 OnReceive 設為 ANY 時，接受最先到達的任何廣播訊號
   第一個觸發後記住該訊號類型（LAST 概念），直到手動重置

4. 指令行數上限（~15 行）→ 設計啟示：
   不強制執行，但複雜度預算條的視覺回饋讓玩家自然傾向簡潔設計
   過度複雜的法陣在介面上會有明確的視覺代價感
```

```
訊號系統：所有訊號為 -100% 到 +100% 的浮點數，1/30 秒一個邏輯 tick
```

### 6. Minecraft Redstone — 訊號強度等級 × 邊緣觸發 × 時序電路

#### 核心機制

```
訊號強度：整數 1–15（0 = 無訊號），非二元 on/off
  Repeater：任意輸入強度 → 輸出恆為 15（放大器）
  Comparator：比較模式（側面 ≥ 後方則輸出 0）/ 減法模式（後方 - 側面）

時序單位：
  1 game tick = 1/20 秒
  1 redstone tick = 2 game ticks（0.1 秒）

Observer（觀察者方塊）：
  偵測正前方方塊狀態變化時輸出 2 game ticks 脈衝
  → 只在「狀態改變」的瞬間觸發，不持續輸出 ← 邊緣觸發的原型

記憶體電路：
  RS Latch：Set/Reset 持久記憶（最基礎的持久狀態）
  T Flip-Flop：每次觸發切換 ON/OFF ← 開關式技能的等效電路
  Monostable：任意長度輸入 → 固定長度單次輸出脈衝 ← 脈衝正規化

Torch Burnout：
  Redstone Torch 在 8 次/秒以上切換後暫時停止輸出（防無限迴圈）
  → 天然的速率限制機制

0-Tick Pulses（已於 1.16 移除）：
  繞過正常時序約束讓訊號同一 tick 激活又停用
  → 設計反例：本系統的施放延遲就是為了防止這種行為
```

#### ▶ 本系統應用

```
1. Observer 邊緣觸發 → 已在 §7 新增「邊緣觸發積木」，與 Dreams 的同一概念互相印證

2. 脈衝正規化（Monostable）→ 已在 §7 新增「單次脈衝積木」
   確保任意長度的持續條件只輸出一次觸發

3. Torch Burnout → 「快速連鎖防護」（加入執行安全機制考量）：
   同一積木序列若在極短時間內被觸發超過 N 次，自動進入短暫冷卻
   防止邊緣觸發被高頻振盪條件無限連續引爆
   類似施放延遲但針對同一積木序列的重複觸發速率

4. T Flip-Flop → 「開關式技能」的技術實作等效電路：
   每次觸發切換 ON/OFF，與藍色圖騰改造刻印「轉為開關式能力」完全對應
   實作時可參考 T Flip-Flop 電路模式

5. 訊號強度 1–15 → 「效果強度等級」概念參考（後期整合）：
   部分刻印效果可定義為多段強度（如「減速 3 級 / 5 級 / 7 級」）
   而非僅有開/關兩態，與 Dreams 的連續訊號值概念一致
   待數值系統確定後統一整合
```

### 1. The Binding of Isaac — 物品協同 × 淚滴修飾器 × 變形系統

#### 協同機制（Item Synergies）

```
Isaac 的協同效果幾乎全部是「湧現式」而非「手動寫死式」：
每個物品只改變淚滴的某個屬性，兩個物品疊加時系統自動計算結果

典型案例：
  Dr. Fetus（淚改砲彈）+ Brimstone（淚改射線）
    → 結果：拋物線飛行的紅色射線炸彈（兩者形態融合）
  Chocolate Milk（蓄力發射）+ Cricket's Body（命中爆散多枚）
    → 結果：蓄力後一次釋放多枚 homing 分裂淚（效果疊加）
  Technology（替換為射線）+ Spoon Bender（淚追蹤 homing）
    → 結果：追蹤彎曲的持續射線

設計關鍵：
  系統不為每對組合單獨設計效果
  各物品修改「淚滴屬性層」的不同欄位，組合結果由屬性合併後統一計算
```

#### 淚滴修飾器（Tear Modifiers）

```
修飾器類型（彼此正交，可疊加）：
  形態類：投射物 / 射線 / 爆炸物 / 旋轉衛星 / 近戰揮擊
  軌跡類：直線 / 拋物線 / 追蹤 homing / 反彈 / 繞行 orbit
  穿透類：穿牆 spectral / 穿敵 piercing / 分裂 splitting
  效果類：引發出血 / 冰凍 / 詛咒 / 吸血 lifesteal / 毒素

正交性保證：大多數修飾器可以無限疊加且互不衝突
衝突例外（手動定義）：
  多種「形態類」修飾器共存時有優先序（射線 > 爆炸物 > 投射物）
```

#### 變形系統（Transformation）

```
16 種角色變形，各自觸發條件（收集 N 個特定類型物品）：

  Guppy（貓咪）：收集 3 個 Guppy 物品 → 飛行 + 攻擊有 50% 機率召喚蒼蠅
                  （Repentance 版本；舊版為每次必定召喚）
  Leviathan（惡魔）：收集 3 個惡魔物品 → 飛行 + 獲得 2 個黑心
  Oh Crap（糞便）：收集 3 個糞便物品 → 周圍產生糞便炸彈
  Spun（藥物）：收集 3 個藥物物品 → +2 傷害 + +0.15 移速

變形觸發時不消耗物品，是額外獎勵層
```

#### ▶ 本系統應用

```
1. 湧現式協同（最重要）→ 刻印屬性層正交設計原則：
   刻印應設計為「修改法陣屬性結構的不同欄位」，而非「觸發獨立效果」
   例：追蹤刻印修改「投射軌跡欄位」，冰凍刻印修改「命中效果欄位」，
       兩者疊加時系統自動合成「追蹤投射物命中冰凍」，無需手動寫這個組合
   設計重點：越多刻印效果透過屬性層合併而非硬編碼，湧現空間越大

```

### 2. Risk of Rain 2 — Proc 係數 × 物品疊加縮放 × 雙刃月球物品

#### Proc 係數（Proc Coefficient）

```
每個技能、物品、容器都有一個 Proc 係數（0.0–1.0+）：
  係數 1.0：完整觸發機率
  係數 0.2：只有 20% 的機率觸發 on-hit 效果
  係數 0.0：完全不觸發任何 on-hit 效果

典型案例：
  Primary（基礎攻擊）：係數通常 1.0
  Loader 的 Knuckleboom（大範圍攻擊）：係數 0.7
  Engineer 的 Sticky Bomb（維持中碰到才計）：係數 0.1
  Captain 的 Orbital Strike 落地範圍：係數 0.1（防止 proc 無限觸發）

Proc 鏈限制（Proc Mask）：
  系統為每次傷害事件維護一個「proc mask」記錄哪些 on-hit 物品已在本鏈中觸發過
  同一 on-hit 物品在同一 proc 鏈中只能觸發一次（不是 proc 係數降為 0）
  例：AtG Missile 命中 → 觸發 Ukulele 閃電 → 閃電不能再觸發 AtG Missile
  → 防止「A → B → A → B」無限交叉鏈，但允許多種不同物品依序觸發一輪
```

#### 物品疊加縮放曲線

```
不同物品採用不同數學曲線，製造不同「第 N 件的效用」感受：

線性疊加（Soldier's Syringe）：
  每件 +7.5% 攻速，疊加 10 件 = +75%，感受線性增長

雙曲線疊加（Tougher Times，格擋機率）：
  公式：f(x) = 1 − 1 / (1 + 0.15x)，x 為疊加件數
  第 1 件：≈13%（in-game 文字描述 "15%"，為公式取近似）
  第 3 件：≈31%（非 45%）
  第 10 件：60%（非 150%，自然趨近上限）

指數疊加（極少數物品）：
  每件都提升倍率，通常被限制在早期疊加才顯著
```

#### 雙刃月球物品（Lunar Items）

```
極強但有顯著代價的物品，以「藍色月亮符文」外觀標識：

  Shaped Glass：傷害 ×2，最大 HP ÷2
  Transcendence：移除所有護盾，改為最大 HP 的 50% 護障（每次戰鬥後回復）
  Brittle Crown：獲得金幣等量傷害（HP 與財富掛鉤）
  Eulogy Zero（每次 tick 有 5% 機率將下一個物品改為月球物品）

設計哲學：
  玩家看到 Shaped Glass 時必須做決策，而非「反正都要拿」
  月球物品讓每次 run 有不同的策略挑戰
```

#### Artifact 系統（全域規則修飾）

```
Artifact 是 run 層面的規則修改，由玩家在開始前主動啟用：

  Artifact of Command：所有物品改為可選（消除隨機性，純策略）
  Artifact of Glass：所有傷害 ×500%，承受傷害 ×500%
  Artifact of Swarms：怪物數量 ×2，大小 ×0.5，HP ÷2
  Artifact of Frailty：墜落傷害造成 HP 50%
  Artifact of Devotion：精英敵人變成寵物，為玩家戰鬥

最多可同時啟用多個 Artifact，效果疊加
```

#### ▶ 本系統應用

```
1. Proc 係數 → 執行容器觸發效率係數（最重要）：
   每種執行容器（投射物/接觸/召喚實體）應有預設 Proc 係數：
     投射物（追蹤型）：0.5（防止高頻追蹤投射物無限觸發 on-hit 刻印）
     接觸型（持續貼身）：0.2 per tick（防止持續接觸的刻印高頻觸發）
     召喚實體攻擊：0.7（接近正常，但低於玩家本體的 1.0）
     玩家本體直接觸發：1.0
   Proc 係數直接填入執行容器設定，邏輯刻印的 on-hit 觸發分支受係數影響
   → 從根本上防止 Proc 鏈無限遞迴，比「積木步數上限」更精確

2. Proc Mask 機制 → VM 執行安全規格（對應 RoR2 proc mask）：
   每次法陣觸發鏈維護一個「已觸發刻印記錄」：
   同一個 on-hit 刻印在同一觸發鏈中只能啟動一次（不是 proc 係數降為 0）
   例：「流血刻印」命中觸發「爆散刻印」，爆散不能再觸發「流血刻印」
   允許不同刻印在鏈中各自觸發一次，防止的是同一刻印無限自觸
   此規則直接加入 §10 執行安全機制

3. 疊加縮放曲線 → 刻印點數投入的非線性收益設計：
   不同類別的刻印應採用不同縮放曲線：
     傷害類刻印：雙曲線（防止無限疊加破壞平衡，自然封頂）
     輔助類刻印（減速、眩暈時長）：線性（疊加感清晰）
     概率類刻印（觸發機率）：雙曲線（防止趨近 100%）
   與 §8 「數值待調整」的未來方向對齊
```

### 3. Hades — Boon 插槽矩陣 × Duo Boon × 熱度懲罰系統

#### Boon 插槽矩陣

```
每位奧林匹斯神祇提供 Boon（恩賜），只能附加在特定插槽上：
  Attack（普通攻擊）
  Special（特殊攻擊）
  Cast（法術投擲）
  Dash（衝刺）
  Call（神祇召喚）

每個插槽同一時間只能持有一個神祇的 Boon
（可替換，但不能雙持）

8 位神祇各自的設計風格（影響 Boon 種類）：
  Zeus（雷電）：鏈式傷害，閃電在多目標間彈跳
  Poseidon（海洋）：擊退，錢幣爆炸
  Athena（神盾）：偏轉，使攻擊反彈敵人投射物
  Ares（戰場）：召喚刀刃，出血持續傷害
  Aphrodite（魅惑）：弱化敵人，強化使用者
  Artemis（狩獵）：暴擊機率，弓矢輔助
  Dionysus（狂歡）：中毒 DoT，多目標感染
  Demeter（大地）：冰霜減速，寒冷疊層
```

#### Duo Boon（雙神協同）

```
當玩家同時持有特定的兩位神祇的至少一個 Boon 時，
有機率在寶藏房出現「Duo Boon」，效果遠比普通 Boon 強力：

  Ares × Athena → Merciful End：偏轉攻擊追加刀刃爆發
  Aphrodite × Artemis → Heart Rend：暴擊對弱化目標傷害大幅提升
  （其餘 26 個 Duo Boon 詳見官方 wiki，共 28 個，涵蓋全部 8 神祇的兩兩組合）

條件：只需持有兩位神祇各至少一個 Boon；無需特定 Boon
出現機率：需兩個親和神祇的條件滿足後，作為稀有寶藏才出現
```

#### Legendary Boon（傳奇恩賜）

```
每位神祇有一個傳奇 Boon，觸發條件：持有該神祇至少 2 個 Tier 1/2 Boon

範例：
  Zeus's Legendary「Splitting Bolt」：需持有 Zeus 至少 2 個一般 Boon
  Artemis's Legendary「Fully Loaded」：需持有 Artemis 至少 2 個一般 Boon

設計意圖：鼓勵「深度投入一位神祇」的專精路線，
與「廣撒多神」路線各有優劣（廣撒更容易解鎖 Duo Boon）
```

#### 武器形態（Weapon Aspects）

```
每件武器有 4 個 Aspect（形態），使用「Titan Blood」解鎖：
  基礎形態（免費）
  3 個傳奇形態（改變武器核心機制）

例：Infernal Arms 劍的四個形態：
  Aspect of Zagreus：標準狀態
  Aspect of Nemesis：每 3 次攻擊第 3 次暴擊
  Aspect of Poseidon（隱藏）：法術可強化普攻，集中使用
  Aspect of Arthur（隱藏）：完全改變武器為聖劍，技能組全部重置

形態選擇不影響可裝備的 Boon，但改變攻擊模式 → 改變最佳 Boon 選擇
```

#### 熱度系統（Pact of Punishment）

```
完成遊戲後，玩家可主動裝備「懲罰條款」提升難度獲取額外獎勵：

條款類型（部分）：
  Hard Labor：敵人傷害 +20%（每點熱度）
  Lasting Consequences：沒有「死亡守護」自動復活
  Jury Summons：大型戰鬥增加精英敵人
  Tight Deadline：每個房間有時間限制
  Extreme Measures：Boss 獲得額外強化技能

熱度等級越高，解鎖的獎勵稀有度越高
條款可自由組合（多條款同時）
```

#### ▶ 本系統應用

```
1. Weapon Aspects → 法陣形態變體（核心設計補充）：
   同一個法陣可定義多個「形態（Aspect）」，每個形態使用相同圖騰骨幹，
   但改變執行方式或核心動作邏輯：
   例：「火焰法陣」
     形態 A（基礎）：投射物軌跡直線，命中燃燒
     形態 B：投射物改為環繞玩家的燃燒衛星 orbit，近戰擴散點燃
     形態 C（傳奇）：投射物消失，改為每次受傷後自動引爆一個燃燒爆炸
   玩家用稀有材料解鎖法陣形態，形態本身不改變刻印槽，但改變基礎行為

2. 熱度系統 → 挑戰條款系統（多人模式延伸）：
   ⚠️ 可供參考，暫不實作（屬後期多人模式功能，核心技能系統完成後再評估）
   玩家可在創建對戰房間時，自行疊加「挑戰條款」增加難度上限，換取更豐厚獎勵：
   例：「禁用持續型能力」「所有 MP 自然回復停止」「隨機 tick 觸發法則逆轉」
   授權機制、獎勵結構待多人模式設計階段確定。
```

### 4. Elden Ring — 灰燼×親和力矩陣 × 狀態積累 × 記憶槽

#### 武器灰燼 × 親和力矩陣（Ash of War × Affinity）

```
每把武器可裝備一個「戰技（Ash of War）」，提供主動技能
同一個戰技配合不同「親和力（Affinity）」，產生完全不同的屬性效果：

例：「Gravitas」戰技（重力吸引球）
  標準親和力：物理傷害，力量/敏捷縮放
  神聖親和力：神聖傷害，信仰縮放（用於打不死族）
  冰霜親和力：物理+冰傷害，附加冰凍積累
  流血親和力：物理+流血積累

同一個戰技 × N 個親和力 = N 種完全不同的定位
親和力需要「謙遜者的磨刀石」才能修改（找到/購買特定道具）
```

#### 狀態積累系統（Status Buildup）

```
6 種主要狀態，均為「積累 → 閾值觸發 → 爆發」機制：

  流血（Bleed / Hemorrhage）：閾值觸發 → 瞬間扣除目標最大 HP 的固定百分比
                              （PvE 約 15%+固定值；確切數字隨 Arcane 屬性縮放）
  冰霜（Frost / Frostbite）：閾值觸發 → -20% 傷害吸收（減傷大幅下降），
                              同時造成固定 % HP 傷害，持續約 30 秒
  毒素（Poison）：閾值觸發 → 長時間低傷害 DoT（持續 90 秒）
  猩紅腐敗（Scarlet Rot）：閾值觸發 → 極高傷害 DoT（持續 60 秒）
  瘋狂（Madness）：只對玩家有效 → 觸發後 FP 歸零並暈眩 2 秒
  睡眠（Sleep）：觸發後無法行動直到受傷

積累設計細節：
  積累量在脫離戰鬥後緩慢衰減
  不同武器（聖杯/獸爪等）附加不同積累量
  同種狀態兩個武器同時積累，積累量直接相加（不平均）
```

#### 記憶槽（Memory Slots）

```
施法需要記憶槽：大多數法術/禱文佔用 1 槽，部分強力法術佔用 2–3 槽
初始 2 槽；收集 8 個記憶之石（+8）+裝備「諾克斯特拉月」護符（+2）= 最多 12 槽
FP 只決定「一次能施放幾次」，記憶槽決定「能裝備哪些法術」

設計張力：
  高 INT 允許使用強力法術，但槽數仍然稀缺
  玩家必須在「帶通用應對組合」vs「帶特定高威力套路」間取捨
```

#### 神奇藥瓶（Flask of Wondrous Physick）

```
可調配的一次性戰前 Buff，由兩種「水晶淚」組合決定效果：

水晶淚範例：
  火焰燃起晶露：FP 消耗降低 25%，持續 3 分鐘
  漫長的活力晶露：最大 HP 臨時提升 20%，持續 3 分鐘
  魔法晶露：法術傷害 +10%，持續 3 分鐘
  迅捷晶露：下一次技能施放速度 ×2

一個瓶子只能裝兩種晶露
晶露在世界各地散落，需要探索收集
使用後 Buff 立即生效，瓶子在休息後補充
```

#### 韌性/姿態系統（Poise/Stance Break）

```
韌性（Poise）：
  決定角色在受攻擊時能否繼續完成動作的抗打斷能力
  不同動作有不同「韌性幀」（只在特定動作幀內有韌性）

姿態崩潰（Stance Break）：
  對敵人造成累計「姿態傷害」超過閾值 → 敵人倒地 → 可處決
  不同攻擊類型（揮擊/刺擊/魔法）姿態傷害不同
  姿態在一段時間後自動回復（閾值重置）

設計意圖：
  讓玩家可以主動設計「打斷組合技」而非單純堆傷害
```

#### ▶ 本系統應用

```
1. Ash of War × Affinity 矩陣 → 圖騰 × 屬性刻印矩陣（最重要）：
   這是本系統圖騰+刻印系統的完美映射：
     圖騰 = Ash of War（決定「做什麼」）
     屬性刻印 = Affinity（決定「用什麼屬性做」）
   同一個圖騰（如「投射圖騰」）配不同屬性刻印（火/冰/電/暗）
   → 完全不同的屬性傷害、觸發碰撞效果和法陣定位
   這意味著玩家設計法陣時，「圖騰選擇」和「屬性刻印選擇」是兩個獨立決策維度，
   設計介面應明確分成兩個選擇步驟，強調這兩個維度的獨立性

2. 狀態積累閾值觸發 → 層疊刻印 + 引爆刻印的設計細化：
   與現有的「層疊刻印（積累層數）+ 引爆刻印（達到 N 層觸發爆發）」直接對應
   進一步細化：
   a. 積累衰減規則：脫離戰鬥（無攻擊/接觸 X 秒後）積累緩慢衰減
   b. 多源積累疊加：同種狀態由多個法陣同時施加時，積累量相加（不取最大值）
   c. 閾值觸發後積累歸零（而非歸 0 後立刻開始下一輪積累，防止秒連觸發）
   以上三條作為「積累類刻印」的通用規格加入 §6 刻印系統

3. 韌性/姿態系統 → 斷招積累閾值（強化 §6 不可打斷刻印設計）：
   現有「斷招刻印」是直接打斷，未來可細化為：
   a. 姿態傷害累計 → 閾值觸發 → 強制打斷（可建造抗性）
   b. 不同類型攻擊有不同姿態傷害係數（確保近戰大招 vs 多段快攻的設計差異）
   c. 姿態恢復時間：被打斷後有短暫的姿態無敵（防止無限連鎖打斷）
   此機制可作為「打斷系統 2.0」在 Phase 2 實作時替換現有簡單版本
```

### 1. Magic: The Gathering — 堆疊系統 × 替代效果 × 層系統

#### 三種能力類型（精確定義）

```
Activated Ability（啟動式能力）：
  格式：[代價] : [效果]
  玩家主動支付代價，能力進入堆疊，對手有機會回應
  → 對應本系統的「宣告型能力」

Triggered Ability（觸發式能力）：
  格式：When / Whenever / At [事件], [效果]
  遊戲事件發生時自動觸發，下一個優先權前進入堆疊
  → 對應本系統的「條件觸發被動」（邏輯刻印的 OnHit / OnDamaged 分支）

Static Ability（靜態能力）：
  格式：陳述句（無代價冒號，無 When/Whenever）
  持續生效，不進入堆疊，建立「持續效果（Continuous Effect）」
  → 對應本系統的「持續生效型能力」
```

#### 堆疊與優先權（Stack & Priority）

```
法術或能力發動 → 進入堆疊（LIFO）
每次一個物件結算後，雙方均可再度發動能力（連鎖機會）

Instant（瞬間）時機：任何時候都可施放
Sorcery（巫術）時機：僅自己的主階段且堆疊為空時施放
→ 兩者時機差異完美對應本系統「宣告型」vs「即時型」的發動限制

Pass Priority：雙方讓渡優先權且堆疊不為空 → 堆疊頂端結算
```

#### 替代效果（Replacement Effects）

```
觸發詞：instead（替代）
機制：監視某事件，以另一事件完全取代，原事件不發生

典型案例：
  「若你會受到傷害，改為受該傷害的一半」（傷害替代）
  「若一個生物會死亡，改為將其放逐」（死亡替代）
  「若你會抽牌，改為不抽」（禁止某事發生）
  「若你會獲得生命，改為失去同量生命」（反轉效果）

替代效果的關鍵特性：
  → 不進入堆疊，即時生效（無法被回應）
  → 多個替代效果同時適用時，受影響的玩家選擇套用哪一個
  → 與「降低傷害」完全不同：減傷 = 數值修改，替代 = 事件替換
```

#### 層系統（Layer System）

```
當多個持續效果同時修改同一個屬性時，7 個 Layer 決定應用順序：
  Layer 1：複製效果
  Layer 2：控制效果
  Layer 3：文字效果
  Layer 4–5：類型、顏色效果
  Layer 6：能力的增加/移除
  Layer 7：數值修改（7a=設定值 → 7b=固定值 → 7c=加減法 → 7d=開關反轉）

同層內依「時間戳記」排序：後生成的效果後應用
依賴性例外（Dependency）：效果 A 影響效果 B 的類型 → 先應用 A

設計意圖：
  確保「無論以什麼順序加入這些效果，最終結果一致」
  消除效果疊加順序不明確導致的裁判爭議
```

#### State-Based Actions（狀態基礎動作）

```
每當任何玩家即將獲得優先權前，系統自動執行一次狀態檢查並同時結算：
  生物 HP ≤ 0 → 死亡
  玩家生命 ≤ 0 → 輸
  效果持續時間到期 → 自動移除
  ...（全部 13 條 SBA）

SBA 的關鍵特性：
  → 同時結算（不依序），不進入堆疊
  → 沒有機會「在某生物死亡前」插入動作（需透過觸發效果才能做到）
  → 確保遊戲狀態永遠是自洽的（不存在「HP = -5 但仍存活」的中間狀態）
```

#### ▶ 本系統應用

```
1. 三種能力類型精確定義 → 確認 §11.6 設計正確：
   MTG 的 Activated / Triggered / Static 與本系統的
   宣告型 / 條件觸發被動 / 持續生效型 完全對應，
   且 Instant vs Sorcery 時機限制與本系統發動規則一致——
   這是外部獨立驗證本系統設計自洽的最強有力參考。

2. 替代效果刻印（全新刻印子類型）：
   新增「替代效果刻印」（建議歸入紅色侵略效果刻印或綠色輔助效果刻印子類）：
   格式：「若 [X事件] 將發生 → 改為 [Y事件]」，原事件完全取消
   與現有刻印的差異：
     傷害刻印 = 修改數值（傷害從 100 改為 80）
     無效化刻印 = 阻止能力宣告進入宣告窗口
     替代效果刻印 = 替換整個事件（「受傷」替換為「回復」）
   範例：
     「若施法者會受到致命傷害 → 改為 HP 剩 1」（死亡替代）
     「若目標會獲得 Buff → 改為該 Buff 轉移至施法者」（效果劫奪）
     「若此法陣效果命中 → 改為對命中者的盟友釋放同等效果」（目標替代）

3. 層系統 → 持續效果疊加順序規範：
   當多個持續生效型效果同時修改同一個屬性時（例如兩個「傷害倍率」持續效果），
   需明確定義應用順序，防止結果因加入順序不同而不一致：
   建議分層：
     L1：類型替換效果（改變能力的基礎類型）
     L2：能力封鎖/授予效果
     L3：數值固定效果（設為固定值，如「傷害固定為 1」）
     L4：加法效果（+X 傷害）
     L5：乘法效果（傷害 ×Y）
     L6：開關反轉效果（正傷害變負）
   同層依「發動時間戳記」排序。

4. State-Based Actions → VM 被動狀態清理鉤子：
   VM 應在每個 tick 的「優先權分配前」執行一輪狀態基礎檢查：
     HP ≤ 0 → 觸發「死亡事件」
     積累層數達閾值 → 觸發「爆發事件」（確保不被其他事件跳過）
     持續效果時間到期 → 自動移除並觸發「效果結束事件」
     Buff 條件失效 → 自動移除依附的規則效果
   SBA 同時處理，不進入連鎖堆疊，確保遊戲狀態永遠自洽。
```

### 2. Guilty Gear Strive — Roman Cancel × Guts × RISC Gauge

#### Roman Cancel（RC）系統

```
消耗 50% Tension（資源槽）可取消當前動作的「恢復幀（recovery frames）」：

四種 RC 類型（依時機區分）：
  紅 RC：攻擊命中瞬間 → 取消恢復幀，目標浮空放緩，可延伸連段
  藍 RC：中立狀態（未攻擊）→ 取消移動慣性，製造位置混亂
  黃 RC：攻擊動作的起手幀 → 取消動作，轉換為其他動作（中斷起手）
  紫 RC：對方防禦中 → 取消恢復幀，使被防禦的動作變安全

核心設計意圖：
  所有動作都有「恢復幀」這個弱點窗口，RC 以資源消耗來規避這個代價
  → 不犯錯（不亂用 RC）vs 善用資源（正確時機 RC 反敗為勝）是技術分水嶺
```

#### Tension Gauge（資源槽）

```
範圍：0–10000（分兩段，各代表 50%）
填充條件：攻擊對手、向對手前進、對手使用 RC（反向填充）
消耗：大部分強力系統行為需消耗 50%（RC / Wild Assault / 部分 Overdrive）

設計特性：積極進攻自然填充資源，防守打法資源效率低
```

#### Guts 系統（HP 生命力保護）

```
HP 越低，受到的傷害越少（乘法縮放）
每個角色有不同的 Guts Rating：數字越低，低血量保護越少

例：某攻擊在滿血造成 100 傷害，在 10% HP 時可能只造成 45–65 傷害
Guts 系統讓「翻盤勝」成為可能，即使大幅落後也維持一線希望
```

#### RISC Gauge（連續受創傷害放大）

```
RISC = R.I.S.C.（Residual Image Shift Count）
連續被命中而不反擊 → RISC 槽填充 → 每個 RISC 等級提升受到的傷害
RISC 槽自然衰減（一段時間不受傷後恢復）

效果：被動挨打者傷害倍率上升，被逼入被動後更難防守
設計意圖：懲罰純防守，鼓勵主動反擊
```

#### ▶ 本系統應用

```
1. Roman Cancel → 「施放中斷換能力」刻印概念：
   新增藍色圖騰改造刻印子類型「快速取消刻印」：
   發動後立即消耗額外 MP（例如：正常消耗的 50%），
   取消此法陣的施放延遲，允許立即接續下一個技能。
   主要用途：製造「本應有後搖的重型能力接連招」的連段策略，
   代價是 MP 顯著提升，非免費功能。
   ⚠️ 是否採納待核心系統完成後評估（與施放延遲機制有互動）

2. Guts 系統 → 低血量傷害衰減（系統層防護）：
   考慮是否設計為全域系統參數：
   當玩家 HP 低於 30% 時，受到的傷害乘以衰減係數（例如 0.7x）
   讓被壓制的玩家有更多反擊機會，減少「一旦落後就完全無力」的挫敗感
   此機制不需要刻印，應為系統預設行為
   ⚠️ 是否採納、衰減閾值與係數待平衡設計時確定

3. RISC Gauge → 「積壓計數器」持續生效型法陣：
   玩家可以設計「若對手連續 N 次受傷未反擊，對其施加傷害放大 Debuff」的法陣，
   以持續生效型 + 計數器積木實現相同效果——
   這確認了現有積木系統已足夠表達 RISC 類機制，無需新增獨立系統。
   RISC 在本系統中是「可創作的效果」，而非內建機制。

4. Tension 積極行動填充資源 → MP 類型設計參考：
   部分 MP 類型（如「戰意 MP」）可設計為「攻擊命中時回復，被攻擊時不回復」，
   鼓勵積極主動的戰鬥風格，與「靈力 MP」（固定回復）形成差異化設計哲學。
   具體 MP 類型設計待 §5 符文系統完善時整合。
```

### 3. Unreal Gameplay Ability System（GAS）— 標籤系統 × 屬性堆疊 × 網路預測

#### 核心架構五組件

```
1. Ability System Component（ASC）：
   每個可使用能力的 Actor 掛載一個 ASC，是所有系統的中心樞紐

2. Attributes + AttributeSet：
   浮點數值集合（HP / MP / 攻擊力 / 速度等）
   結構：BaseValue（基礎值）+ CurrentValue（含修改器後的即時值）

3. GameplayTags（遊戲標籤）：
   層級式字串標籤（例：「Status.Debuff.Frozen」是「Status.Debuff」的子標籤）
   用於表示任何遊戲狀態，是所有組件之間共同的「語言」

4. GameplayEffects（遊戲效果）：
   能力對屬性和標籤的修改器
   三種持續時間：Instant（立即一次性）/ Duration（固定時長）/ Infinite（手動移除）
   疊加策略：聚合（加總）/ 覆蓋 / 最大值 / 忽略（按設定）

5. GameplayAbilities（遊戲能力）：
   定義能力的完整邏輯，包含 Cost Effect（消耗）/ Cooldown Effect（冷卻）
   可異步執行（透過 AbilityTasks 分解為子任務）
```

#### GameplayTags 機制詳解

```
標籤層級：「Status.Debuff.Frozen」自動繼承父標籤「Status.Debuff」和「Status」
          查詢「Status.Debuff」會同時匹配「Status.Debuff.Frozen」

能力的標籤設定（每個 Ability 可設定）：
  Required Tags（必要標籤）：Actor 必須持有這些標籤才能啟動此能力
  Block Tags（封鎖標籤）：Actor 持有這些標籤時，無法啟動此能力
  Activation Required Tags：效果等同 Required，用於更細節情境
  Cancel Abilities With Tags：啟動時自動取消 Actor 身上持有這些標籤的能力

典型應用：
  「Debuff.Stun」標籤 → 封鎖所有設定了 Stun 為 Block Tags 的能力
  「Buff.EnhancedFire」標籤 → 解鎖設定了 EnhancedFire 為 Required Tags 的能力
```

#### 屬性修改堆疊（Attribute Modifier Stack）

```
Final Value = (Base + ΣAdditive) × ΣMultiplicative

計算順序固定：
  1. 取 Base Value
  2. 加上所有 Additive 修改器（加法）
  3. 乘以所有 Multiplicative 修改器（乘法）

設計意圖：
  避免「先加後乘」vs「先乘後加」導致的數值歧義
  無論以什麼順序加入修改器，Final Value 計算結果一致
```

#### 網路預測（Network Prediction）

```
問題：多人遊戲中，玩家動作和伺服器結算有延遲

GAS 解法：Prediction Key 系統
  1. 客戶端立即本地執行能力（Local Predicted）——看起來即時有反應
  2. 同時將 Prediction Key 送至伺服器，伺服器驗證
  3. 若伺服器結果與客戶端一致 → 確認，繼續
  4. 若不一致 → 伺服器強制回滾客戶端狀態（Rollback）

預測範圍：Attributes、GameplayTags、GameplayCues 均可預測
不可預測：Spawn 新 Actor（需伺服器執行）
```

#### ▶ 本系統應用

```
1. GameplayTags 層級標籤系統（最重要架構參考）：
   建立正式的能力標籤系統，取代目前「文字描述式」的封鎖規則：
   標籤結構範例：
     Ability.Fire（所有含火元素的能力）
     Ability.Sustained（所有持續生效型能力）
     Status.Debuff.Frozen / Status.Buff.Haste
     Element.Fire（角色持有火屬性）
   
   每個法陣設定：
     Required Tags：「需要持有 Element.Fire 才能發動此法陣」
     Block Tags：「若持有 Status.Debuff.Silence 則無法發動任何 Ability.* 法陣」
   持續生效型封鎖的實作方式變為：施加「Block.Ability.Fire」標籤 → 
   所有設定了 Block.Ability.Fire 為 Block Tags 的法陣自動無法發動
   → 比現有「文字判斷封鎖」更系統化、更容易實作

2. 屬性修改分層計算 → VM 屬性修改積木的計算規範：
   所有修改屬性的積木遵循固定順序：
     Base → +加法修改器總和 → ×乘法修改器總和
   設計介面顯示「Base Value / Total Additive / Multiplicative 」三欄，
   讓玩家清楚看到能力點投入在哪一層

3. GameplayEffects 持續時間類型 → 已整合至 §6 效果持續時間分類（四種）：
   本系統採用即時型 / 定時型 / 脫戰型 / 永久型，詳見 §6。

4. Network Prediction → 宣告型能力的客戶端預測：
   本系統的 50ms tick + 宣告窗口設計中，可採用 GAS 的 Prediction Key 模式：
   宣告型能力在客戶端立即播放起手動畫（本地預測），
   伺服器確認後才正式結算效果；若伺服器拒絕（例如 MP 不足），
   客戶端回滾動畫——讓玩家感受即時反饋，同時維持伺服器權威性。
   此為 §11.6 多人同步機制的具體實作參考。
```

### 4. Warcraft 3 世界編輯器 — 物件繼承 × 觸發器三段式 × 虛擬單位

#### 物件編輯器（Object Editor）

```
所有能力、單位、Buff 均以「繼承自基礎物件」的方式創建：
  建立自訂能力 → 選擇基礎能力（如 Storm Bolt、Blink、Aura 等）
  → 自訂物件只覆蓋與原版不同的欄位，其餘繼承原版值

欄位類型：
  整數欄位（技能等級、次數）
  浮點欄位（傷害值、冷卻時間、施法距離）
  字串欄位（技能名稱、圖示路徑、工具提示）
  布林欄位（是否可以被無視、是否可學習）
  物件引用欄位（改變施法動畫、Buff 效果）

執行期修改（JASS）：
  SetAbilityRealLevelField（abilityId, level, field, value）
  允許觸發器在遊戲運行中動態修改技能數值
```

#### 觸發器編輯器（Trigger Editor）

```
每個觸發器由三段組成：
  Event（事件）：何時觸發 — 「某單位施放某技能時」「玩家離開遊戲時」「計時器到期」
  Condition（條件）：過濾 — 「施放的技能等於 Fireball」「施法者屬於玩家 1」
  Action（動作）：執行 — 「建立虛擬單位」「對目標施加效果」「播放音效」

觸發器可以互相呼叫，形成複雜的邏輯鏈
GUI 觸發器可轉換為 JASS 程式碼（相當於從 Scratch 模式切換到 JASS 模式）
```

#### 虛擬單位系統（Dummy Unit）

```
WC3 中許多效果靠「不可見的虛擬單位」實現：
  1. 在目標位置建立一個不可見、不可選取的虛擬單位
  2. 讓虛擬單位施放對應的技能到真正的目標
  3. 效果執行完後立即移除虛擬單位

用途：實現非常規施法者施法、對目標位置觸發特效、附著 Buff 到目標

Buff 系統：
  能力命中後對目標附加 Buff（Buff 是一個持續存在的效果容器）
  Buff 可以有週期效果（每 0.5 秒造成傷害）、到期效果、被移除效果
  Buff 的存在允許持續效果「感知目標是否還活著」
```

#### 多等級能力數值（Per-Level Data）

```
每個技能最多 3 個等級（可擴展），每等級可有獨立數值：
  Level 1：傷害 75、冷卻 8 秒
  Level 2：傷害 150、冷卻 7 秒
  Level 3：傷害 225、冷卻 6 秒

升級消耗技能點（Skill Points），單位每升 2 級獲得 1 點技能點
```

#### ▶ 本系統應用

```
1. 物件繼承 → 圖騰/刻印模板繼承系統（架構設計參考）：
   圖騰和刻印在實作時應設計為「繼承自基礎模板，只儲存差異欄位」：
     「冰凍命中刻印」繼承「命中效果刻印」，只覆蓋「效果類型 = 冰凍」欄位
     「追蹤投射物圖騰」繼承「投射物圖騰」，只覆蓋「軌跡類型 = 追蹤」欄位
   優點：大幅減少資料重複，新增刻印變種時只需指定差異部分
   這是 §6「刻印屬性層正交設計」的技術實作層的直接參考

2. 觸發器三段式（Event → Condition → Action）→ 確認 §7 VM 設計：
   WC3 的觸發器與本系統的邏輯刻印積木結構完全對應：
     Event = 觸發積木（When HP < 30%、On Hit、On Timer）
     Condition = 條件積木（如果「目標是敵人」）
     Action = 效果積木（施放火球、施加 Buff）
   已在計畫中；WC3 是這套架構成熟可行的最有力實例驗證。

3. 虛擬單位 → 確認執行容器設計（§4）：
   WC3 的 Dummy Unit 機制與本系統「執行容器」是同一個設計模式：
   「不可見的施法代理人執行效果」= 本系統的投射物/接觸容器裝載法陣執行。
   兩者均是「效果的載體」而非「效果本身」。

4. 執行期數值修改（SetAbilityRealLevelField）→ 不採納：
   WC3 此功能是地圖編輯器的開發者工具，非玩家面向的設計機制。
   本系統的對應需求（根據戰況動態強化法陣效果）已由
   「邏輯刻印持久變數 + 條件積木 + Buff/Debuff 刻印」組合實現，
   不需要底層的法陣欄位直接覆寫積木。

5. Buff 作為持續效果容器 → §4 接觸容器的設計補充：
   WC3 的 Buff 模型直接對應本系統「接觸容器」的持續效果行為：
     Buff 存在期間 → 接觸容器持續掛載在目標上
     Buff 週期效果 → 接觸容器的「每 X 秒執行一次」觸發
     Buff 移除事件 → 「效果結束時」觸發的邏輯積木
   建議接觸容器設計時明確定義這三個生命週期鉤子（掛載/週期/移除）
```

### 能力分解與內化系統（待規劃）

> ⚠️ 此為後期內容，設計前需先確立「技能開發等級」或「熟悉度」機制。

```
概念：
  玩家可以將取得的預製能力或道具能力，分解為能力點，
  用於重新設計自己的法陣。

分解門檻（需同時滿足）：
  a. 技能開發等級達到閾值
     玩家整體的法陣設計熟練度（設計過越多複雜法陣，等級越高）

  取得 → 使用（積累熟悉度）→ 理解（熟悉度閾值）→ 分解內化 → 重新設計
```

### 法陣形態變體（Aspect）（參考用）

> 現階段不實作。保留於此供未來構思 Synergy 時參考。  
> 「我感覺它可以幫助我以後構思 Synergy，未來或許會用到。」  
>  
> 主要懸案：①刻印共用跨形態時的語義衝突尚無解決方案；②條件形態依賴物品系統，待物品系統設計後才可實作。

```
同一個法陣可定義多個「形態（Aspect）」，各形態使用相同的圖騰骨幹，
但改變圖騰的核心執行行為：

  基礎形態：免費，法陣建立時即存在
  傳奇形態：需稀有材料解鎖（每個法陣最多 3 個傳奇形態）

特性：
  → 形態切換不改變刻印槽（刻印可跨形態共用）
  → 形態切換在戰鬥外自由進行，切換後才生效
  → 同一法陣的不同形態共享法陣名稱，但可各自附加副標題

範例：「火焰投射法陣」
  形態 A（基礎）：直線投射，命中燃燒
  形態 B（傳奇）：投射物改為環繞玩家的燃燒衛星 orbit，近戰命中擴散點燃
  形態 C（傳奇）：取消投射物，改為每次受傷後自動引爆一個燃燒爆炸

設計意圖：
  相同刻印搭配在不同形態下可以有完全不同的使用方式；
  形態切換成本（稀有材料）使這成為有重量感的選擇，而非免費換皮。
  靈感來自 Hades 的 Weapon Aspect 系統。
```

#### 條件形態（Conditional Aspect）

現有形態（Aspect）需玩家在戰鬥外主動切換。條件形態是擴充類型，**由系統自動根據條件切換**，無需玩家手動操作：

```
設定方式：
  在法陣設計介面中，將某個傳奇形態標記為「條件形態」
  並填寫啟動條件（觸發後自動覆蓋當前形態）

支援的啟動條件類型：
  裝備條件：持有指定物品 / 道具時啟動（靈感來自 Dota 2 Aghanim's Scepter）
           → ⚠️ 依賴物品系統，待物品系統設計後再實作
  狀態條件：持有指定 Tag / 處於指定狀態時啟動
           → 範例：HP < 30% 時切換為「死亡意志形態」
  外部條件：持有特定稱號 / 達成特定里程碑時啟動

行為：
  → 條件成立時，法陣自動切換至條件形態（無視當前手動選擇的形態）
  → 條件不再成立時，自動回退到條件成立前的形態
  → 同一法陣只允許設定一個條件形態
  → 條件形態可以與基礎形態或傳奇形態共存（條件形態優先序最高）
```

> ⚠️ 「裝備條件」類型的實作依賴物品系統與裝備欄設計，規格待物品系統確立後補齊。
